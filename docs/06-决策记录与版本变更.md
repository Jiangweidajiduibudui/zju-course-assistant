## 1. 产品决策(D01–D34)

> 决策日期均为 ≤2026-07-08(以《产品文档修改指导》确认为准);状态均为**有效**。

### D01 · 规则优先(部分冻结已由 D30 解除)
- **原问题**:选课录取规则大量细则未核实,原文档却以"档位占位(稳/悬/危)"先行实现命中率与顺位设计。
- **采用**:完整规则核实前,冻结志愿、顺位、录取风险及依赖规则的推荐设计;不采用临时"双层志愿"数据模型;不用"稳/悬/危"作无规则依据的占位符。
- **放弃**:v1.2/v1.3 的"档位占位先行"方案。
- **影响**:02(冻结声明)、04 §3.3、01 FR-6/FR-7。
- **后续状态**:D30 已确认第一/二轮志愿数据模型与基本顺位规则，解除这两部分的冻结;具体录取概率、风险档位和校准仍冻结。

### D02 · 独立实现
- **原问题**:原文档误认 Lazuli 为宽松协议、"可复用代码"(实为 GPLv3,见 F-01)。
- **采用**:TypeScript 独立重写;Lazuli 与 Quantum 仅作参考;不复制 Lazuli 或校内 Quantum 代码;参考公开 Quantum(MIT)时保留许可证声明。
- **放弃**:复用 Lazuli 代码。
- **影响**:01 §3、04 §7。
- **复审条件**:若未来决定开源本项目且考虑 GPL 兼容,再评估。

### D03 · chalaoshi 使用方式
- **原问题**:数据授权未取得,是否作为上线前置。
- **采用**:Demo 后端按需直接抓取并缓存 chalaoshi;长期可按 D32 建立镜像数据层与评论向量索引;明确来源、原页链接、投诉/删除/停止收录机制;不宣称拥有数据;**接受尚未完全消除的合规风险**。
- **放弃**:把站方授权作为上线前置条件。
- **影响**:03 §3.4、05 §7。
- **复审条件**:收到站方或作者异议;或产品扩大分发规模时。

### D04 · LLM 数据发送范围
- **采用**:首次引导统一征得同意;隐私声明分别说明用户主动导入的课程/基线数据会提交给网站后端处理，以及调用 LLM 时可向用户自配端点发送完整课表、偏好和对话;姓名、学号、Cookie、会话令牌和 API key 永不进入提示词;第三方供应商可能保存请求数据。Demo 不在服务端持久化用户规划数据。
- **影响**:01 FR-11、04 §5.1。
- **复审条件**:新增数据类别进入提示词时。

### D05 · 推荐核心 = 确定性求解器
- **原问题**:原 M4 设计为 Gather-Act-Verify agent loop;但本任务步骤固定、候选有限、硬约束可机检。
- **采用**:确定性程序负责组合生成、硬约束、冲突检测、最小扰动和最终校验;LLM 负责偏好结构化、评论理解、合法方案软排序和解释。
- **放弃**:agent loop(含"只有 M4 是真正 Agent"预设,见 F-06)。
- **影响**:04 §3/§4、01 §4-3。
- **复审条件**:仅当 benchmark 证明 agent loop 明显更优。

### D06 · 第一版范围
- **采用**:第一版公开发布即完整 beta,不先公开阉割版;内部仍按模块逐步实现验收;**规则未核实期间不发布**。
- **说明**:"冻结规则相关设计"(D01)约束的是**开发顺序**,"完整 beta"约束的是**发布范围**——两者不矛盾:非规则依赖模块先行开发,发布等待规则闭环。
- **影响**:01 §10、05 §6(G0)。
- **复审条件**:选课窗口时间压力导致取舍时,由产品负责人重新决策。

### D07 · 结果表述
- **采用**:"最优课表"→"基于当前候选和偏好的推荐方案 / 当前规则下的相对最优方案";"真实命中率"→"录取风险估计";显示依据、数据时间和不确定性;无法可靠评估时显示"暂不可评估"。
- **影响**:01 §8(对外文案)、全部 UI 文案。
- **复审条件**:风险算法上线并通过校准后,措辞可再评估。

### D08 · session 模型
- **采用**:session 只由用户手动创建;新 session 以当时 zdbk 状态为基线、从零开始;待选池/规则/方案/对话不自动继承;历史 session 可查阅但不自动恢复。
- **放弃**:v1.1 的跨轮默认继承;同时**撤销 v1.3 文档中"第一轮未中→下一轮几乎不可能中"的因果论证**(未经证实,不作为任何设计依据,见 F-08)。
- **影响**:01 FR-1、04 §2.1。
- **复审条件**:真实使用数据显示跨轮复用需求强烈时。

### D09 · 页面集成(已被 D31 取代)
- **历史决策**:曾允许 content script 注入 zdbk 页面，并要求 DOM/CSS 隔离和兼容性回归。
- **当前状态**:产品已改为独立网站，不再注入或修改 zdbk 页面;业务写接口禁令仍保留。

### D10 · LLM 兼容
- **采用**:允许任意 OpenAI 兼容端点;配置时检测连通性、结构化输出、超时和基本能力;能力不足的端点不得开放完整推荐。
- **影响**:04 §4.1、01 FR-10。

### D11 · API key
- **采用**:默认长期保存于网站客户端存储;明确提示本地明文存储风险;UI 仅掩码;不进日志;不做服务端持久化;提供替换和一键删除。如因 CORS 需要后端代理，key 只能在单次请求内使用，不得存储或记录。
- **影响**:04 §4.2、01 FR-10。

### D12 · 评价摘要
- **采用**:使用**近五年**评论;必须同时总结优点和缺点;<5 条不生成;5–9 条生成但显著标记"低样本";≥10 条正常;评论经时间筛选、去重和长度限制;始终保留原页链接,注明评论通常非本课程专属。
- **放弃**:v1.3 的"近三年"口径。
- **影响**:01 FR-3、04 §4.3-②、05 §4。

### D13 · 教师匹配
- **采用**:姓名和学院**同时一致**才自动匹配;同名、学院缺失或冲突时禁止猜测;用户确认后本地保存映射;支持更正和删除映射。
- **影响**:01 FR-3、05 §1。

### D14 · 均绩匹配
- **采用**:只有课程名规范化后唯一匹配、或经用户确认,才展示本课程均绩;无法确认时显示"暂无可可靠匹配的本课程均绩";**不展示其他课程均绩作为替代**。
- **放弃**:v1.3 的"对不上则显示总体并标注非本课程"。
- **影响**:01 FR-3、05 §1。

### D15 · beta 发布门槛(八项)
- **采用**:八项确认后可直接发布公开 beta,不要求先封闭试用(逐项见 [05](05-验收与测试计划.md) §6;发布另受 D06 规则前置约束)。
- **影响**:05 §6。

### D16 · 推荐数量
- **采用**:每次只输出一份方案;按用户偏好顺序给出当前输入下的相对最优方案;用户可在结果上继续调整并重新优化。
- **放弃**:多方案对比输出。
- **影响**:01 FR-6。

### D17 · 规则优先级
- **采用**:`手动锁定 > 局部硬约束 > 全局硬约束 > 局部软偏好 > 全局软偏好`;无解时不自动放松;不允许 LLM 决定牺牲哪条硬约束;展示冲突来源由用户修改。
- **影响**:01 FR-5、04 §3.2。

### D18 · 取消拖拽,改为基线补全
- **原问题**:原 M5 以"拖拽摆课表、拖入即第一志愿"为核心交互;但志愿结构未核实,且拖拽语义与真实录取机制耦合过深。
- **采用**:删除拖拽规划与"拖入即第一志愿";输入 = 当下 zdbk 已选结果(固定)+ 已填志愿及顺序(锁定)+ 待选池 + 偏好;输出 = 补齐后的完整推荐方案;产品不决定用户该选什么课程。
- **D30 后的状态**:志愿结构已确认，推荐方案可给出指向具体教学班的建议顺位;仍不恢复拖拽代填，由用户手动录入 zdbk。
- **放弃**:v1.3 的手动拖拽规划工作台。
- **影响**:01 §6/FR-6/FR-7、04 §3.1。
- **复审条件**:如真实使用证明需要更富的志愿编辑交互，可重新评估(须新决策)。

### D19 · zdbk 状态同步
- **采用**:session 创建时保存用户输入/导入的基线快照;支持重新导入后的手动同步;展示新增/删除/顺位变化差异;用户确认后才更新锁定基线并重新优化未锁定部分;不自动覆盖当前方案。网站不在 Demo 中监听 zdbk 页面变化。
- **影响**:01 FR-1、04 §2.2。

### D20 · 关键数据刷新
- **采用**:生成推荐、重新优化和最终确认前，刷新网站后端可获取的 chalaoshi/公开数据;对无法自动刷新的 zdbk 导入数据显示导入时间并要求用户确认;数据变化后重新计算;锁定方案失效时必须提示用户。
- **影响**:01 FR-6/FR-8、03 §2.5、04 §2.3。

### D21 · 无 key 降级
- **采用**:无 key 或 LLM 不可用时**不生成推荐**;仍可浏览课程、评分、均绩、点名信息和待选池;评价摘要、偏好理解、推荐和解释禁用。
- **放弃**:v1.3 的"无 key 仍可纯手动排课"(该路径随拖拽工作台一并取消,见 D18)。
- **影响**:01 FR-12、04 §4.6。

### D22 · key 配置时机
- **采用**:首次启动不强制配置;首次点击评价摘要或推荐时进入配置向导;取消后返回信息功能。
- **影响**:01 FR-10/FR-11(与 D23 的关系:隐私同意在首次启动,key 配置延后——同意≠配置)。

### D23 · 隐私同意
- **采用**:首次引导一次性说明并取得客户端存储、用户主动导入数据的后端处理、chalaoshi 后端抓取和未来 LLM 发送授权;不同意则无法进入产品;设置页提供隐私说明和客户端数据清除入口。
- **影响**:01 FR-11、04 §5.3。

### D24 · 评论提示注入
- **采用**:仅通过系统提示词声明评论是不可信数据;不增加专门的数据隔离层;**接受此防护较弱的风险**;评论摘要功能不得修改待选池、规则或方案状态。
- **影响**:04 §5.2、05 §4/§7。
- **复审条件**:出现实际注入事件,或摘要功能获得任何状态通道时。

### D25 · 推荐输出校验
- **采用**:固定 JSON Schema;Schema、教学班 ID、课程覆盖、锁定状态和约束全部校验通过后才能展示;失败时不从自然语言中猜测结果;LLM 输出不能直接写入 zdbk。
- **影响**:04 §4.3、01 FR-12。

### D26 · 调用预算
- **采用**:产品不限制 LLM 调用次数、超时、token 或费用;由用户配置的端点控制;设置页提醒费用和长时间等待风险。

### D27 · 生成过程
- **采用**:展示阶段进度;允许取消;取消、失败或校验不通过时不修改当前方案;完整成功后一次性应用;旧请求结果不能覆盖用户的新操作。
- **影响**:04 §4.4、01 FR-6。

### D28 · 缓存
- **采用**:Demo 由网站后端按需直接抓取 chalaoshi，教师详情和评论缓存 3 天;显示更新时间并支持强制刷新;使用过期缓存时显著标记。长期数据层按 D32 演进。
- **影响**:03 §3.2、01 FR-3。

### D30 · 第一版选课规则模型
- **原问题**:志愿分组、顺位处理、课程优先级、待选人数口径和容量释放规则未闭环，D01 因此冻结了相关设计。
- **采用**:第一版只建模第一、二轮概率筛选;优先级只按课程面向对象判定，不再细分、不考虑学分影响;同一课程最多 3 个志愿、同一时间段最多 3 个志愿，两类约束叠加，每个顺位指向具体教学班;筛选按“课程面向对象 → 志愿顺序 → 同档随机”处理;待选人数包含三个志愿档，第一版对各档使用同一聚合口径;预置课退掉后当轮不释放容量，下一轮释放。
- **不在范围**:第三/四轮、补选、低分定点选课、额外次级优先顺序、精确录取概率与风险档位。
- **影响**:02 §1–§6、04 §2/§3、05 §1/§3;志愿模型与基本顺位规则解冻，风险数值继续冻结。
- **复审条件**:第一/二轮抓包或官方新规则与本口径冲突;或产品决定扩展到第三/四轮。

### D31 · 产品形态改为网站
- **原问题**:Chrome 扩展对安装、权限、浏览器差异和 zdbk 页面结构有强依赖，不利于快速演示和迭代。
- **采用**:产品改为独立 Web 网站，由 Web 前端承载课程浏览、待选池、规则、课表、推荐与对话，由后端 API 承载数据抓取、解析、求解与强制同源 LLM 代理。
- **zdbk 边界**:独立网站不能默认复用用户在 zdbk 的登录 Cookie。Demo 不索取或保存统一身份认证密码、Cookie 或会话令牌，使用手动输入/导入或脱敏 fixture 构造课程与基线数据。长期 zdbk 接入方式单独决策。
- **不变原则**:advise-only;网站不调用选课、退选、调序等 zdbk 业务写接口。
- **取代**:D09 页面注入方案;原 Side Panel、content script、extension service worker 和 `chrome.storage.*` 架构退出现行设计。
- **影响**:README、01 全部用户旅程与 FR-1/FR-2/FR-11、03 zdbk 鉴权边界、04 §1/§2/§4、05 E2E 与兼容性测试。

### D32 · chalaoshi 数据获取与长期数据层
- **Demo**:由网站后端按用户请求直接抓取 chalaoshi 的教师索引、教师详情和评论接口/页面;必须限频、缓存、超时、保留来源链接并明示抓取失败。
- **长期方向**:考虑建立可定期更新的镜像数据层，保存原始快照和规范化结构;评论文本可建立向量索引，用于语义检索、去重和摘要取样。
- **数据源原则**:向量数据库只是派生索引，不是事实源;原始快照/规范化记录才是可追溯数据源，每条数据保留来源 URL、抓取时间和内容版本。
- **合规边界**:延续 D03;不声称拥有数据，提供来源链接、删除/停止收录机制和快照过期策略。
- **未决事项**:镜像数据层的更新频率、存储位置、是否公开镜像站、embedding 模型和删除同步 SLA 另行决策；向量数据库起步方案已由 D33 选定为 PostgreSQL + pgvector。
- **影响**:01 FR-3/FR-12、03 §3、04 §1、05 §1/§2/§5/§7。

### D33 · 正式技术栈
- **目标**:优先满足 AI 开发语料充足、生态成熟、性能较强、边界清晰和技术种类少。
- **采用**:Node.js 24 LTS + TypeScript 单 package 模块化单体；React + Vite 构建前端，Fastify 承载同源 REST API 与生产静态托管；Dexie/IndexedDB 保存用户状态；PostgreSQL 保存公共抓取缓存；`pg` 执行参数化 SQL；Zod 作为前后端和 LLM Schema 来源；Cheerio 解析上游 HTML；L1 LRU + L2 PostgreSQL 缓存；确定性求解器保持纯 TypeScript。
- **长期数据层**:评论向量索引优先 PostgreSQL + pgvector，不单独部署向量数据库；Demo 不启用向量扩展。
- **不采用**:Next.js App Router/Pages Router、pnpm workspace、Drizzle/Prisma、Redis、微服务、Bun/Elysia、Python/Go/Rust 后端、独立向量数据库和 Playwright 线上抓取。
- **项目状态**:技术栈已定稿，进入工程脚手架与首个纵向切片开发阶段；O-Q1 关闭。
- **影响**:README、04 §1/§7、05、07。

### D34 · 版本与第三方 API 治理
- **采用**:所有直接依赖精确锁定稳定版本，不使用 `^`、`~`、`latest`、beta、rc、canary 或 experimental API；版本基线和易错 API 由 [07](07-技术栈与版本基线.md) 维护。

- **升级**:业务 PR 不顺手升级依赖；每次只升级一个框架族，独立 PR 完成迁移、类型检查、测试、构建和性能回归。
- **文档治理**:新增第七份权威文档《技术栈与版本基线》；旧技术选型提案归档。

### D35 · 正式 Demo 与六人开发计划
- **采用**:不设置对外 mock 版 walking skeleton；一次直接跑通正式 Demo。团队内部仍采用持续合并、fixture 契约测试和隐藏未完成入口。
- **范围**:正式 Demo 必须覆盖导入/合成数据、学分上限、待选池、chalaoshi 抓取或 seed fallback、两阶段 LLM、Top10 候选、志愿提交方案、预期课表投影、导出和无 zdbk 写请求证明。
- **影响**:新增 [08-开发实施计划](08-开发实施计划.md)。

### D36 · 选课数学模型集中模块
- **采用**:所有课程/时间志愿组、时间槽、考试冲突、学分上限、缺失硬字段、候选枚举、Top10、课表投影、最小扰动和最终校验集中在纯 TypeScript `selection-model` 模块。
- **边界**:前端只渲染，后端只编排，LLM 只排序和解释；任何模块不得复制一份简化硬约束逻辑。
- **影响**:04 §3、08 §6。

### D37 · 时间、考试与志愿组冲突口径
- **采用**:时间槽只区分春/夏/秋/冬学期、星期几、第几节，不区分单双周。上课时间重叠不直接判无解，交给志愿预排序与 LLM 软判断；考试时间重叠是硬约束，不同课程同一考试时间直接判无解；同一课程不同教学班考试时间重叠视为正常。
- **缺失处理**:考试时间缺失的教学班标记未知，停留在待选池，不参与排课。
- **志愿组优先级**:同一课程有 2 个及以上教学班时形成课程志愿组；课程组优先于时间槽志愿组。若时间槽组与课程组冲突，时间槽组失效。
- **影响**:02 §4/§6、04 §2/§3、08 §6。

### D38 · 学分上限与缺失学分
- **采用**:每个 session 必须由用户填写学分上限；第一版不设置课程门数上限。
- **缺失处理**:教学班学分字段缺失时停留在待选池并说明原因，不参与排课。
- **边界**:学分不进入录取优先级、概率或风险估计。
- **影响**:02 §4/§6、04 §3、08 §6。

### D39 · 两阶段 AI 排序与 Top10
- **采用**:LLM 先对每个志愿组内教学班排序；`selection-model` 按该顺序生成 Top10 完整候选方案；LLM 再比较完整候选方案；最后由 `selection-model` 终校验。
- **缺失软指标**:教师 GPA、评分、评论、点名等软指标缺失保持中性，由 LLM 综合判断，不用字典序或固定扣分代替。
- **影响**:04 §3/§4、05 §4、08 §7。

### D40 · LLM 强制同源代理与 SSRF 防护
- **采用**:所有 LLM 请求必须通过同源 Fastify 后端代理；前端不直连模型端点。用户 key 只随单次请求进入后端内存，不保存、不记录、不写日志。
- **安全边界**:仅允许 HTTPS；阻断 localhost/private/link-local/reserved IP；DNS 与重定向后均复核；限制端口、请求体、超时、并发和响应大小。
- **影响**:04 §4/§5、08 §7。

### D41 · Demo seed cache 与公开仓库数据边界
- **采用**:Demo 可提交结构忠实但内容合成的 chalaoshi seed cache；live 抓取成功时优先展示真实数据，失败时显式降级到演示数据。
- **禁止**:不把真实 chalaoshi 评论批量提交到公开 GitHub；不复制 Lazuli 或校内 Quantum 代码。
- **影响**:03 §3、05 §2/§7、08 §3/§14。

### D42 · zdbk-like UI 与 AI 友好日志
- **采用**:UI 模仿 zdbk 的“待筛选志愿”和“预期课表”心智模型，但不写入 zdbk。日志采用稳定 JSON Lines，包含 requestId、generationId、module、action、duration、errorCode 等字段，适合 AI 维护。
- **隐私边界**:默认无远程遥测；日志和诊断导出不记录课程详情全文、偏好全文、评论原文、LLM I/O、key、Cookie、学号姓名。
- **影响**:01 FR-7/FR-12、04 §6、08 §8/§12。
