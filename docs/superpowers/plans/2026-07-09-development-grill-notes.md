# 2026-07-09 开发计划 Grilling 结论记录

> 本文记录 `$grill-with-docs` 追问后由产品负责人确认的开发约束。已合并进最终开发文档：[docs/08-开发实施计划.md](../../08-开发实施计划.md)。

## 1. Demo 与协作方式

| 主题 | 确认结论 |
|---|---|
| Demo 形态 | 不做“先跑通假的 walking skeleton 再换真模块”的对外路径；一次直接跑通正式 Demo。 |
| 团队合并 | 仍允许持续合并、夹具契约测试、隐藏未完成入口；主分支必须保持可构建、可测试，不允许 mock 伪装为正式功能。 |
| 开发范围 | 当前阶段聚焦功能实现规划，不创建代码脚手架。 |
| 负责人职责 | 负责人掌握正式 Demo 主线、契约、集成门禁与最终演示；5 名组员按模块边界并行。 |

## 2. 选课数学模型

| 主题 | 确认结论 |
|---|---|
| 模型集中 | 课程/时间志愿组、考试冲突、学分上限、候选枚举、Top10 方案、课表投影、最终校验集中在一个纯 TypeScript `selection-model` 模块。 |
| 课程志愿组 | 同一课程出现 2 个及以上教学班时形成课程志愿组；课程组内教学班不能再参与跨课程时间槽志愿组。 |
| 时间槽志愿组 | 仅在没有被课程志愿组占用时成立；如果与课程志愿组冲突，优先服从课程志愿组，时间槽志愿组失效。 |
| 时间粒度 | 区分春/夏/秋/冬学期、星期几、第几节；不区分单双周。 |
| 上课时间重叠 | 不作为硬无解；进入志愿预排序和 LLM 判断。 |
| 考试时间重叠 | 同一时间段的不同课程考试重叠为硬无解；同一课程不同教学班考试重叠视为正常。 |
| 考试时间缺失 | 标记未知，停留在待选池，不参与排课。 |
| 学分上限 | 用户必须填写学分上限；不设置课程门数上限。教学班学分缺失时停留在待选池并说明原因。 |

## 3. AI 排序与推荐生成

| 主题 | 确认结论 |
|---|---|
| LLM 路径 | 所有 LLM 请求走同源 Fastify 后端代理；前端不直连模型端点。 |
| key 处理 | 用户 key 只随单次请求传给后端；后端不保存、不记录、不写日志。 |
| 排序方式 | 采用两阶段 AI：LLM 先对每个志愿组内教学班排序；确定性模型按该顺序生成合法组合；LLM 再比较完整候选方案；最后由确定性模型终校验。 |
| 候选规模 | 完整方案候选截断为 Top10。 |
| 缺失软指标 | GPA、评分、评论等软指标缺失保持中性，不用字典序或固定扣分替代 AI 综合判断。 |
| 硬字段缺失 | 考试时间、学分等会影响硬校验的字段缺失时，不进入排课并给出说明。 |

## 4. UI、导入、数据与日志

| 主题 | 确认结论 |
|---|---|
| UI 参考 | 模仿 zdbk 选课网站的“预期课表”和“待筛选志愿”心智模型；不写入 zdbk。 |
| 输出双层 | 一层是可手动录入的志愿提交方案；另一层是当前首选投影出的预期课表。 |
| 导入范围 | 正式 Demo 支持内置 Demo 数据、指定 JSON 导入、UI 少量字段修正、导出当前 JSON；不做 Excel、截图 OCR、复制 zdbk HTML、自动登录。 |
| chalaoshi | Demo 优先直接抓取 chalaoshi；缓存为空或抓取失败时使用结构忠实但内容合成的 Demo seed cache；不把真实评论提交到公开 GitHub。 |
| SSRF 防护 | LLM 代理仅允许 HTTPS；阻断 localhost/private/link-local/reserved IP；DNS 与重定向都需复核；限制端口、body、超时和并发。 |
| 日志 | JSON Lines 结构化、字段稳定、适合 AI 读取；默认无远程遥测；不记录课程详情、偏好、评论原文、LLM I/O、key、Cookie、学号姓名。 |

## 5. 仍需在开发中复核的边界

- 录取概率、风险档位和校准方法仍不进入第一版。
- zdbk 长期数据接入方式仍另行决策；正式 Demo 不自动登录、不调用 zdbk 写接口。
- 公共 GitHub 开源许可证尚未决定；在决定前不复制 GPL 或校内未授权代码，不提交真实评论数据。
