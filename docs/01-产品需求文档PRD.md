## 1. 背景与问题

浙江大学本科选课的特点:

- **高度自由**:培养方案给出必修,四年内修完即可毕业;学生自主组合课程。
- **多阶段**:预置(学校预先置入,选课开始前完成)→ **随机选课阶段(本产品服务的阶段)** → 补选(申请制)。
- **顺序志愿录取**:详细录取规则以 [02-规则核对档](02-浙大选课规则核对档.md) 为唯一权威;其中大量细则尚未核实。
- **选课系统**:`https://zdbk.zju.edu.cn/`(教务 jwglxt,选课模块 xsxk)。

**学生真实的选课流程(本产品要服务的)**:

1. 看预置课表 + 查老师评价,退课定初版;
2. 明确想上哪些课(**课程级决策,归用户,产品不介入**);
3. 每门课里排除"绝对不选"的教学班(综合老师、给分、时间、地点);
4. 在候选教学班里权衡时间/地点/评价/给分/待选人数/余量,排出一份自己满意、可执行的方案。

**痛点**:第 3、4 步耗费大量精力。现有工具停留在"信息展示"层,没有一个走到"决策辅助"层。

> ⚠️ 待补:以上流程描述来自开发者自身经验与竞品观察,**尚无正式用户研究证据**。beta 前应至少做小样本访谈验证(开放问题,见 06 §O-Q)。

## 2. 目标用户

浙江大学在读本科生(普通用户,不假设技术背景;"自备 LLM key"对其是显著门槛,故配置向导是关键路径功能,见 FR-10)。

## 3. 竞品与差异化

| 工具 | 能做到 | 天花板 |
|---|---|---|
| **Lazuli**(浏览器插件,GPLv3) | 在 zdbk 页面内联 chalaoshi **评分数字** + 链接 + "待选/余量"难度比 | 只有一个评分数字;纯信息展示 |
| **Quantum**(桌面端;公开仓库 MIT 仅为框架壳,选课逻辑在校内仓库、保留所有权利) | 登录 + 拉课程 + 心愿单 + 课表 + 快照趋势 + 一个简单命中概率 | chalaoshi 仍只取评分数字;无 AI、无排序;需安装桌面程序 |

**共同天花板**(基于公开功能观察):① chalaoshi 只被当作"一个评分",按课均绩与评价文本未被利用;② 都停留在信息展示,未进入决策辅助。

**我们的差异化(一句话)**:

> 从"信息层"跨到"决策辅助层"——读懂 chalaoshi(评分 + **按课均绩** + **近五年评价摘要**),在用户自己圈定的候选范围内,给出一份**基于当前候选、规则和偏好的相对最优推荐方案**,并支持在其上持续调整与重新优化。

**合规立场**(权威决策见 06 · D02/D03):Lazuli 为 **GPLv3**,本项目采用 **TypeScript 独立重写**,Lazuli 仅用于参考公开交互、数据结构与接口事实,不复制代码;公开 Quantum(MIT)可参考架构,使用时保留许可证声明;校内 Quantum 仅参考机制,禁止复制代码;chalaoshi 数据使用方式见 D03,项目不宣称拥有数据,并接受尚未完全消除的合规风险。

## 4. 核心产品原则

1. **尽可能真实**:数据尽可能准确;无法确认时明确标注"缺失 / 需确认 / 暂不可评估",绝不呈现可能错误的信息。信任是产品命脉。
2. **全程 advise-only**:插件**永不调用选课、退选、调序等业务写接口**,只输出"推荐方案",由用户手动录入 zdbk。
3. **确定性求解器保硬约束,LLM 只做模糊语义**:组合生成、硬约束、冲突检测、最小扰动、最终校验由确定性程序完成——在**已验证规则和输入数据范围内**进行确定性校验;偏好结构化、评论理解、合法方案软排序、解释归 LLM(D05)。
4. **页面增强不破坏 zdbk 原有功能**:允许 content script 注入与页面修改,DOM/CSS 隔离命名空间并做兼容性回归(D09);主体验仍在侧边栏。
5. **规则优先**:选课规则未核实的部分,产品**不装懂**——相关功能冻结或显示"暂不可评估"(D01、D07)。
6. **对话为辅但有实权且不黑箱**:LLM 的任何"改状态"动作,先转成可见的结构化改动、由用户确认后生效。

## 5. 产品范围与边界

| 维度 | 在范围内 ✅ | 不在范围内 ❌ |
|---|---|---|
| 决策层级 | **教学班级别**(同一门课选哪个班、如何补齐方案) | **课程级别**(该上哪些课)——归用户 |
| 阶段 | **随机选课阶段** | 预置执行、补选(申请制) |
| 写操作 | 无(advise-only) | 代选/代填/代提交/代退/代调序 |
| 附加模块 | 评价信息、待选池、推荐补全、课表可视化 | 毕业审查(因人而异,已删)、余量提醒(轮内不变,已删)、趋势预测/导出分享/资料收集(留接口,later) |

**待选池的严格定义**:待选池只表达两件事——用户希望**补齐哪些课程**,以及每门课**接受哪些候选教学班**。产品不建议池外班级,不建议用户该上什么课。

## 6. 端到端用户旅程

> 步骤 0(仅首次):**首次引导 + 隐私同意**——产品导览 + 隐私声明,一次性取得本地读取/存储与未来 LLM 发送的授权;不同意则无法进入产品(D23)。**首次启动不要求配置 LLM key**(D22)。

1. **开 session**:用户手动新建 session;插件读取当下 zdbk 状态(已选上的教学班 + 已填志愿)保存为**基线快照**(D08、D19)。
2. **搜课**:侧栏内复用 zdbk 自身搜索(分类动态读取 + 选课搜索引擎),浏览课程与教学班。
3. **看班**:点课程 → 教学班按老师归类 → 看评分/按课均绩/点名信息,按需生成近五年评价摘要(FR-3)。
4. **入池**:把候选教学班加入待选池(支持批量),按课程分组管理(FR-4)。
5. **设偏好**:调整规则栏(作用范围 + 偏好顺序 + 硬约束);或对话表达模糊诉求 → 确认生成规则(FR-5)。
6. **生成推荐**:点"生成推荐" → 强制刷新关键数据(D20)→ 确定性求解器 + LLM 软排序,输出**一份**补齐后的完整推荐方案(D16、D18);过程可取消,显示进度(D27)。
7. **调整**:锁定/更换候选/移除 → 点"重新优化"做最小扰动重排(变更高亮,可回滚)(FR-7)。对话可辅助微调(确认制)。
8. **定稿**:点"确认"锁定方案 → 进入填写阶段。
9. **手动录入 zdbk** → 回来点"检查":重新拉取 zdbk 实际状态,与定稿方案逐条一致性校验,指出差异;只提示,不代填(FR-8)。
10. **下一轮**:再手动新建 session,以新基线从零开始;历史 session 可查阅,需要的部分(如规则栏)手动取回,不自动继承(D08)。

## 7. 功能需求(FR)

> 每条 FR 含:职责 / 输入 / 输出 / 异常 / 验收标准(AC)。AC 与 [05-测试计划](05-验收与测试计划.md) 一一映射。接口细节一律引用 [03](03-数据接口契约.md),技术方案引用 [04](04-AI与技术架构设计.md)。

### FR-1 · session 与基线

- **职责**:session 由用户手动创建;创建时保存 zdbk 基线快照(已选教学班 + 已填志愿);支持基线同步。
- **输入**:用户"新建 session"操作;zdbk 只读接口(见 03)。
- **输出**:session(基线快照、待选池、规则栏、方案、对话历史、状态历史栈);历史 session 列表(只读查阅 + 手动取回单项)。
- **异常**:未登录 zdbk → 提示登录,阻塞需要 zdbk 数据的功能;非选课期 → 明示,信息类功能仍可用。
- **基线同步(D19)**:支持手动同步,也可检测页面变化;展示新增/删除/顺位变化差异;**用户确认后**才更新锁定基线并重新优化未锁定部分;不自动覆盖当前方案。
- **AC-1.1** 新建 session 后基线快照与 zdbk 实际状态一致;**AC-1.2** 历史 session 不自动恢复到新 session;**AC-1.3** 基线差异必须经确认才生效。

### FR-2 · 课程搜索与浏览

- **职责**:侧栏内复用 zdbk 自身的搜索能力浏览课程/教学班。
- **输入**:分类(**动态读取,不硬编码**,见 03)或课名关键词。
- **输出**:课程列表 → 教学班列表(教师、时间地点、考试、余量/容量、待选人数等,字段语义以 03 为准)。
- **单向桥接**:侧栏可读取 zdbk 页面当前搜索上下文,一键在侧栏重现同一搜索。
- **异常**:接口失效/结构变更 → 明示"数据获取失败",不静默展示旧数据。
- **AC-2.1** 分类列表来自运行时读取;**AC-2.2** 搜索结果与 zdbk 页面同条件结果一致。

### FR-3 · 教师评价卡片(差异化的根)

- **职责**:为每个教学班的每位老师提供 chalaoshi 评分、按课均绩、点名信息、近五年评价摘要、原页链接。
- **展示**:教学班按老师归类;一班多师则每位老师各显示一行;显示数据更新时间;缓存过期时显著标记(D28)。
- **教师匹配(D13)**:姓名和学院**同时一致**才自动匹配;同名、学院缺失或冲突时**禁止猜测**,交用户确认;确认后本地保存映射,支持更正与删除。未命中 → 明示"chalaoshi 暂无该老师数据"。
- **均绩匹配(D14)**:只有课程名规范化后唯一匹配、或经用户确认,才展示"本课程均绩";无法确认时显示"暂无可可靠匹配的本课程均绩",**不用其他课程均绩替代**。
- **评价摘要(D12)**:取**近五年**评论(经时间筛选、去重、长度限制);**必须同时总结优点和缺点**;少于 5 条不生成;5–9 条生成但显著标记"低样本";≥10 条正常生成;始终保留原页链接,并注明评论通常不是本课程专属;摘要按需生成(点击触发),可重新生成。
- **异常**:chalaoshi 不可达/限流 → 降级为只显示已缓存数据或"暂无",不阻断其他功能;无 LLM key → 摘要入口显示"配置 LLM 后解锁",点击进入 key 向导(FR-10)。
- **AC-3.1** 同名教师场景不自动匹配;**AC-3.2** 均绩不跨课程替代;**AC-3.3** 评论少于 5 条不生成摘要;**AC-3.4** 摘要同时含优点与缺点;**AC-3.5** 原页链接始终可达。

### FR-4 · 待选池

- **职责**:承载用户策展的候选教学班(以教学班为单位,按课程分组)。
- **交互**:单个 `+` 入池;一门课的班级列表支持多选/全选**批量入池**;支持单个或**批量移除**;自动去重。
- **边界**:严格只定义"目标课程 + 候选教学班";推荐器**只在池内决策**,绝不引入或提示池外班级。
- **AC-4.1** 批量入池/移除正确且去重;**AC-4.2** 任何推荐方案中不出现池外教学班。

### FR-5 · 偏好表达(规则栏)

- **职责**:结构化表达偏好,供求解器与 LLM 使用。
- **结构**:一叠规则栏;默认预置 1 条全局规则栏,可添加。每条 = **作用范围**(全局/某课/某类)+ **偏好顺序**(拖动排序,如 均绩>评分>时间)+ **硬约束**(禁止/必须,如"绝不早八")。
- **优先级(D17,固定)**:`手动锁定 > 局部硬约束 > 全局硬约束 > 局部软偏好 > 全局软偏好`。
- **无解时**:不自动放松规则;不允许 LLM 决定牺牲哪条硬约束;展示冲突来源,由用户修改。
- **对话补充**:LLM 可把模糊诉求("想轻松点")转成**可见的规则栏建议**,用户确认后生效。
- **AC-5.1** 优先级次序在求解中严格生效;**AC-5.2** 无解时呈现冲突来源且不自动放松;**AC-5.3** 对话生成的规则未经确认不生效。

### FR-6 · 推荐生成(基线补全)

- **职责**:在当前基线之上,补齐一份完整推荐方案。
- **输入(D18)**:当下 zdbk 已选结果(固定)、已填但待筛选的志愿及顺序(锁定)、待选池、用户偏好。
- **输出(D16)**:**每次只输出一份方案**——基于当前候选和偏好的推荐方案;附依据、数据时间与不确定性说明(D07)。
- **过程(D20、D27)**:生成前强制刷新余量/待选人数/正式状态并显示更新时间;展示阶段进度;允许取消;取消/失败/校验不通过时不修改当前方案;完整成功后一次性应用;旧请求结果不能覆盖用户新操作。
- **录取风险(D01、D07)**:规则与算法未闭环前,一律显示"**暂不可评估**";不用任何无依据档位或数字。
- **异常**:LLM 不可用/校验失败 → 见 FR-12。
- **AC-6.1** 已选教学班在方案中不被改动;**AC-6.2** 已填志愿及顺序保持锁定;**AC-6.3** 方案无硬约束冲突;**AC-6.4** 取消/失败不产生半成品状态;**AC-6.5** 风险栏在规则闭环前恒为"暂不可评估"。

### FR-7 · 方案调整与重新优化

- **职责**:在推荐结果上持续调整。**不提供拖拽规划**(D18 已取消);调整通过结构化操作完成。
- **交互**:课表视图(锚视图,可折叠/全 panel 展开)展示方案;对方案条目可 **锁定/解锁**、**在同课程候选内更换教学班**、**移除**;任何改动后出现"**重新优化**"按钮 → 最小扰动重排(锁定项不动,其余尽量少动)+ 变更高亮;维护状态历史栈,支持**回滚到上一状态**。
- **志愿顺序编辑**:冻结——志愿分组与顺位规则未核实前不提供(见 02 冻结声明)。
- **AC-7.1** 锁定项在重新优化中不变;**AC-7.2** 重新优化后变更被高亮;**AC-7.3** 回滚恢复到上一个完整状态。

### FR-8 · 定稿与一致性检查(advise-only 最后一公里)

- **职责**:"确认"锁定方案 → 填写阶段(用户手动录入 zdbk)→ "检查"做一致性校验。
- **检查**:重新拉取 zdbk 实际状态,与定稿方案逐条比对,指出漏选/顺序不符/选错班;**只提示,不代填**;锁定方案因数据变化失效时必须提示(D20)。
- **AC-8.1** 校验能识别漏选/错班/顺序差异;**AC-8.2** 全流程无任何写接口调用。

### FR-9 · 对话助手

- **职责**:① 模糊诉求 → 可见结构化改动(确认制);② 解释推荐;③ 回答课程/老师问题。
- **原则**:为辅但有实权;任何改状态动作先转结构化改动、用户确认;评论摘要等内容性输出**不得**修改待选池/规则/方案状态(D24)。
- **session 操作**:「新规划」「历史」以界面按钮为主入口;`/new`、`/resume` 保留为等价快捷命令。
- **AC-9.1** 对话不经确认无法改变任何状态;**AC-9.2** 解释引用的均为方案内真实数据。

### FR-10 · LLM key 配置向导(关键路径)

- **职责**:让普通本科生顺利配好自备 key。触达差异化功能(摘要/推荐)的实际开关。
- **时机(D22)**:首次启动**不**强制配置;首次点击评价摘要或推荐时进入向导;取消后返回信息功能。
- **向导**:厂商一键预设(DeepSeek/通义等,base URL 与推荐 model 自动填)→ 申请 key 图文教程深链 → 粘贴后自动检测:连通性、结构化输出、超时与基本能力(D10);**能力不足的端点不得开放完整推荐**;失败给出可读原因。
- **key 管理(D11)**:默认长期保存至本地存储;明确提示本地明文存储风险;UI 仅显示掩码;不进日志;提供替换与一键删除。
- **AC-10.1** 能力检测不通过时推荐功能不可用且原因可读;**AC-10.2** key 全程不出现在日志与界面明文中;**AC-10.3** 一键删除后本地无 key 残留。

### FR-11 · 首次引导与隐私同意

- **职责**:首次启动进入产品导览,并一次性取得授权(D23):本地读取 zdbk 登录态、 本地存储、未来向用户自配 LLM 端点发送数据。
- **隐私声明必须说明(D04)**:同意后调用 LLM 时会发送**完整课表、偏好和对话**;姓名、学号、Cookie、会话令牌和 API key **永不进入提示词**;**第三方模型供应商可能保存请求数据**;其余数据不出本机。
- **不同意则无法进入产品**;设置页提供隐私说明与本地数据清除入口。
- **AC-11.1** 未同意前无任何功能可用;**AC-11.2** 声明文案覆盖 D04 全部要点;**AC-11.3** 数据清除入口有效。

### FR-12 · 产品级降级行为

| 情形 | 行为 |
|---|---|
| 未登录 zdbk | 提示"请先登录 zdbk 选课系统";依赖 zdbk 的功能等待登录 |
| 非选课期 / 数据为空 | 明示"当前非选课期或无可选数据";评价查询等信息类功能仍可用 |
| chalaoshi 不可达/限流 | 只显示缓存或"暂无",标注数据时间;不阻断其他功能 |
| 教师/均绩匹配不确定 | 按 D13/D14:交用户确认或如实标缺失 |
| **无 key 或 LLM 不可用(D21)** | **不生成推荐**;仍可浏览课程、评分、均绩、点名信息和待选池;评价摘要、偏好理解、推荐和解释禁用 |
| LLM 输出 Schema 校验失败(D25) | 不从自然语言中猜测结果;按失败处理,不改动当前方案 |
| 断网 | 用本地缓存 + 明示离线与数据时间 |

- **AC-12.1** 每种情形有明确 UI 状态,绝不假装成功;**AC-12.2** 无 key 路径下信息类功能完整可用。

## 8. 对外文案规范(禁止表述)

| ❌ 禁止 | ✅ 使用 |
|---|---|
| 真实命中率 | 基于当前数据和已确认规则的录取风险估计 |
| 最优课表 | 当前候选、规则和偏好下的相对最优推荐方案 |
| 工具保正确 | 在已验证规则和输入数据范围内进行确定性校验 |
| (风险)稳/悬/危 等无依据档位 | 暂不可评估(规则与算法闭环前) |
| 零页面注入 | 页面增强不得破坏 zdbk 原有功能 |

所有结果展示必须附:**依据、数据更新时间、不确定性说明**(D07)。
