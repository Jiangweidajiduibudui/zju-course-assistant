## 1. 修改目标

现有文档包含有价值的产品洞察，但混合了产品愿景、外部事实、规则假设、接口记录、AI 架构和测试计划。修改后的文档体系必须达到以下目标：

1. 将事实、假设、产品决定和实现方案分离。
2. 删除或修正已经被外部核验否定的描述。
3. 不把未确认的选课规则写成实现契约。
4. 保持 advise-only：插件不调用选课、退选、调序等业务写接口。
5. 采用确定性求解器作为推荐核心，LLM 只承担模糊语义任务。
6. 让每项功能都有明确输入、输出、异常状态和验收标准。
7. 第一版公开发布为完整 beta，但内部仍按模块逐步实现和验证。

## 2. 必须拆分出的六份文档

### 2.1 产品需求文档（PRD）

只负责：

- 目标用户与问题；
- 产品价值和边界；
- 用户旅程；
- 功能需求；
- 产品级降级行为；
- 对外文案；
- 成功指标；
- 发布范围。

不得在 PRD 中维护具体接口字段、抓包细节、Prompt 或算法代码。

### 2.2 浙大选课规则核对档

只负责：

- 选课阶段和轮次；
- 志愿组、志愿顺序和录取规则；
- 本专业、辅修等优先级；
- 时间冲突、考试冲突、学分和门数限制；
- 每条结论的来源、核验时间、核验方式和置信度；
- 尚未确认的开放问题。

志愿、顺位、录取风险及相关推荐设计必须等本文件完成后再重新设计。

### 2.3 数据接口契约

分别记录 zdbk 和 chalaoshi：

- 端点、方法和必要参数；
- 当前实测字段；
- 字段语义与解析规则；
- 示例响应（删除个人信息）；
- 缓存和刷新策略；
- 登录失效、非选课期、空数据、接口变更等异常；
- 只读接口与禁止调用的写接口；
- 最近核验日期。

### 2.4 AI 与技术架构设计

只负责：

- Chrome 扩展架构；
- content script、Side Panel、service worker 的职责；
- 状态模型；
- 确定性求解器；
- LLM 输入输出 Schema；
- Prompt、模型能力检测、取消和错误处理；
- API key 存储；
- 隐私、安全与提示注入风险；
- 日志和可观测性。

### 2.5 验收与测试计划

只负责：

- 单元测试；
- 固定接口样例；
- 求解器正确性测试；
- LLM 固定评测集；
- 端到端测试；
- zdbk 兼容性回归；
- beta 发布门槛；
- 已知未覆盖风险。

### 2.6 决策记录与版本变更

每项决定至少包含：

- 决策编号；
- 日期；
- 原问题；

- 放弃方案；

## 3. 外部核验后必须修正的事实

### 3.1 Lazuli 许可证

原文“公开、宽松协议，可复用代码”错误。

- Lazuli 实际使用 GPLv3，不是宽松许可证。
- 本项目采用 TypeScript 独立重写。
- Lazuli 仅用于参考公开交互、数据结构和接口事实，不复制代码。
- 公共 Quantum 的 MIT 架构可参考，使用时保留许可证声明。
- 校内 Quantum 仅参考机制，禁止复制代码。

参考：

- https://github.com/ADSR1042/Lazuli
- https://github.com/ADSR1042/Lazuli/blob/master/LICENSE
- https://github.com/XmmShp/Quantum

### 3.2 chalaoshi 接口与数据立场

- `search.json` 当前可用。
- 教师详情页当前可读取评分、评分人数、点名比例和按课程均绩。
- 评论接口当前可直接返回 HTML。
- 原文所写 `/getsession` 当前返回 404，不得继续写成必要步骤。
- 评论作者仍享有内容版权；不得写“chalaoshi 没有数据版权”或“其声明必然无效”。
- 产品负责人决定继续离线打包教师索引并缓存评论，不把取得站方授权作为上线前置条件；此项属于已知合规风险。

必须保留的控制措施：

- 明确数据来源和原页链接；
- 提供投诉、删除和停止收录机制；
- 不宣称拥有数据；
- 教师详情和评论缓存 3 天；
- 显示更新时间并支持强制刷新；
- 使用过期缓存时明确标记。

参考：

- https://chalaoshi.de/
- https://chalaoshi.de/static/json/search.json
- https://github.com/zjuchalaoshi/chalaoshi

- 公开项目只能作为交叉参考，不能替代真机抓包和固定样例。

### 3.4 Agent 方法论

## 4. 已确认产品决策

### D01：规则优先

- 完整选课规则核实前，冻结志愿、顺位、录取风险及依赖这些规则的推荐设计。
- 不采用临时“双层志愿”数据模型。
- 不用“稳/悬/危”作为无规则依据的占位符。

### D02：独立实现

- 使用 TypeScript 重写。
- Lazuli 和 Quantum 仅作为参考。
- 不复制 Lazuli 或校内 Quantum 代码。

### D03：chalaoshi 使用方式

- 继续离线打包教师索引并缓存评论。
- 明确来源、原页链接、投诉、删除和停止收录机制。
- 不宣称拥有数据。
- 接受尚未完全消除的合规风险。

### D04：LLM 数据发送范围

- 首次引导统一征得同意。
- 同意后可向用户自配 LLM 端点发送完整课表、偏好和对话。
- 姓名、学号、Cookie、会话令牌和 API key 永不进入提示词。
- 隐私声明必须说明第三方模型供应商可能保存请求数据。

### D05：推荐核心

- 不采用原 Agent loop。
- 确定性程序负责组合生成、硬约束、冲突检测、最小扰动和最终校验。
- LLM 负责自然语言偏好结构化、评论理解、合法方案软排序和解释。
- 只有 benchmark 证明 Agent loop 明显更优时才重新评估。

### D06：第一版范围

- 第一版公开发布即完整 beta，不先公开阉割版。
- 内部开发仍按模块逐步实现和验收。
- 规则未核实期间不发布。

### D07：结果表述

- “最优课表”改为“基于当前候选和偏好的推荐方案”或“当前规则下的相对最优方案”。
- “真实命中率”改为“录取风险估计”。
- 显示依据、数据时间和不确定性。
- 无法可靠评估时显示“暂不可评估”。

### D08：session

- session 只由用户手动创建。
- 每个新 session 以当时 zdbk 状态为基线，从零开始。
- 待选池、规则、方案和对话不自动继承。
- 历史 session 可查阅，但不自动恢复到新 session。

### D09：页面集成

- 不再限制“零注入”。
- 允许 content script 注入和修改页面，只要不破坏原功能。
- DOM 和 CSS 必须隔离命名空间并进行兼容性回归。
- advise-only 不变：禁止调用选课、退选、调序等业务写接口。

### D10：LLM 兼容

- 允许任意 OpenAI 兼容端点。
- 配置时检测连通性、结构化输出、超时和基本能力。
- 能力不足的端点不得开放完整推荐。

### D11：API key

- 默认长期保存到 `chrome.storage.local`。
- 明确提示本地明文存储风险。
- UI 仅显示掩码。
- 不进入日志，不使用 `chrome.storage.sync`。
- 提供替换和一键删除。

### D12：评价摘要

- 使用近五年评论。
- 必须同时总结优点和缺点。
- 少于 5 条不生成。
- 5 至 9 条可生成，但显著标记“低样本”。
- 10 条及以上正常生成。
- 评论需经过时间筛选、去重和长度限制。
- 始终保留原页链接，并注明评论通常不是本课程专属。

### D13：教师匹配

- 姓名和学院同时一致才自动匹配。
- 同名、学院缺失或冲突时禁止猜测。
- 用户确认后本地保存映射。
- 支持更正和删除映射。

### D14：均绩匹配

- 只有课程名规范化后唯一匹配，或经用户确认后，才展示本课程均绩。
- 无法确认时显示“暂无可可靠匹配的本课程均绩”。
- 不展示其他课程均绩作为替代。

### D15：beta 发布门槛

确认以下八项后可直接发布公开 beta，不要求先进行封闭试用：

1. zdbk 登录态读取、搜索、课程和教学班数据正常；
2. 不调用选课、退选、调序等写接口；
3. 教师与均绩匹配符合规则；
4. 待选池、规则、求解、调整、回滚和一致性检查全流程可完成；
5. 输出方案无已知硬约束冲突；
6. LLM 失败时按产品定义降级；
7. API key 删除、缓存清理和隐私同意有效；
8. 没有阻断使用的已知错误。

### D16：推荐数量

- 每次只输出一张方案。
- 根据用户偏好顺序给出当前输入下的相对最优方案。
- 用户可以在推荐结果基础上继续调整并重新优化。

### D17：规则优先级

固定为：

```text
手动锁定
> 局部硬约束
> 全局硬约束
> 局部软偏好
> 全局软偏好
```

- 无解时不自动放松规则。
- 不允许 LLM 擅自决定牺牲哪条硬约束。
- 展示冲突来源，由用户修改。

### D18：取消拖拽，改为基线补全

- 删除拖拽规划和“拖入即第一志愿”。
- 输入为当下 zdbk 已选结果、已填志愿、待选池和用户偏好。
- 已正式选上的教学班固定。
- 已填但待筛选的志愿及顺序锁定。
- 待选池定义用户希望补齐的课程及候选教学班。
- 产品不决定用户应该选什么课程。
- 输出为补齐后的完整推荐方案。

### D19：zdbk 状态同步

- session 创建时保存基线快照。
- 支持手动同步，也可检测页面变化。
- 展示新增、删除和顺位变化等差异。
- 用户确认后才更新锁定基线并重新优化未锁定部分。
- 不自动覆盖当前方案。

### D20：关键数据刷新

- 生成推荐、重新优化和最终确认前强制刷新余量、待选人数和正式状态。
- 显示更新时间。
- 数据变化后重新计算。
- 锁定方案失效时必须提示用户。

### D21：无 key 降级

- 无 key 或 LLM 不可用时不生成推荐。
- 仍可浏览课程、评分、均绩、点名信息和待选池。
- 评价摘要、偏好理解、推荐和解释禁用。

### D22：key 配置时机

- 首次启动不强制配置。
- 首次点击评价摘要或推荐时进入配置向导。
- 取消后返回信息功能。

### D23：隐私同意

- 首次引导一次性取得本地读取、存储和未来 LLM 发送授权。
- 不同意则无法进入产品。
- 设置页提供隐私说明和本地数据清除入口。

### D24：评论提示注入

- 当前决定仅通过系统提示词声明评论是不可信数据。
- 不增加专门的数据隔离层。
- 接受此防护较弱的风险。
- 评论摘要功能不得修改待选池、规则或方案状态。

### D25：推荐输出

- 使用固定 JSON Schema。
- Schema、教学班 ID、课程覆盖、锁定状态和约束全部校验通过后才能展示。
- 失败时不从自然语言中猜测结果。
- LLM 输出不能直接写入 zdbk。

### D26：调用预算

- 产品不限制 LLM 调用次数、超时、token 或费用。
- 由用户配置的模型端点控制。
- 设置页提醒费用和长时间等待风险。

### D27：生成过程

- 展示阶段进度。
- 允许取消。
- 取消、失败或校验不通过时不修改当前方案。
- 完整成功后一次性应用。
- 旧请求结果不能覆盖用户的新操作。

### D28：缓存

- 教师索引随插件版本或后台机制更新。
- 教师详情和评论缓存 3 天。
- 显示更新时间并支持强制刷新。
- 使用过期缓存时显著标记。

### D29：文档体系

- 采用本指导第 2 节定义的六份文档。
- 禁止在多份文档中重复维护同一事实。

## 8. 交给 Claude 的评审要求

请 Claude 以资深产品经理、AI 产品工程师和技术文档审校者三种视角进行只读评审，重点回答：

1. 已确认决策之间是否仍有逻辑冲突？
2. “冻结规则相关设计”与“第一版完整 beta”是否被正确区分为开发顺序和发布范围？
3. 新的“基线补全”流程是否完整取代了原拖拽课表流程？
4. 待选池是否仍严格只决定目标课程和候选教学班？
5. 确定性求解器与 LLM 的职责是否没有重叠或空白？
6. 是否仍存在把未经核验的 zdbk 字段、规则或接口写成事实的情况？
7. 是否准确处理 GPLv3、校内 Quantum 和 chalaoshi 的合规边界？
8. 隐私声明是否覆盖本地长期保存 API key、发送完整课表和第三方端点留存风险？
9. 无 key、能力检测失败、生成取消、Schema 失败、接口失效等状态是否有明确产品行为？
10. 每项核心需求是否能够映射到验收测试？

Claude 不应：

- 擅自修改已确认决策；
- 用常识补全未核实的浙大选课规则；
- 重新引入 Agent loop、拖拽课表或多方案输出；
- 将公开可访问等同于无版权或无合规风险；
- 在规则未闭环前声称文档已经可以完整实现。

建议 Claude 的输出格式：

```text
1. 阻断性问题
2. 重要逻辑问题
3. 文档协作问题
4. 对每份目标文档的修改建议
5. 尚需产品负责人决定的问题
6. 总体是否具备重写条件
```

## 9. 本指导的完成标准

后续重写完成时，应能够证明：

- 六份文档职责无重叠；
- 所有已确认决策均已落入唯一权威文档；
- 所有开放规则仍明确标为未确认；
- 原文已被外部核验否定的事实全部修正；
- 产品旅程已从“拖拽规划”改为“读取当前状态并补齐”；
- Agent loop 已替换为确定性求解工作流；
- beta 八项门槛可以逐项测试；
- 没有新增未经产品负责人确认的关键决定。
