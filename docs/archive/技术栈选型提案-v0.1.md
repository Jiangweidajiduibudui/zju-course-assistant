## 1. 结论摘要

推荐采用 **Node.js 24 LTS + pnpm workspace + Next.js App Router + TypeScript** 的模块化单体：一个 Web 应用同时承载界面和同源后端 API，领域模型、求解器、抓取器和数据访问以 workspace package 隔离。

| 层级 | 推荐技术 | 用途 |
|---|---|---|
| 运行时与包管理 | Node.js 24 LTS、pnpm | 统一前后端运行时；锁文件保证复现 |
| Web 框架 | Next.js App Router、React、TypeScript strict | 网站 UI、Route Handlers、服务端渲染与同源 API |
| UI | Tailwind CSS 4、按需引入 shadcn/ui、Lucide | 快速搭建可维护的工作台界面；组件代码归仓库所有 |
| 输入/契约校验 | Zod | 导入文件、API、LLM 结构化输出共用 Schema |
| 客户端持久化 | Dexie + IndexedDB | session、基线、待选池、规则、方案历史、教师映射和 LLM 配置 |
| 服务端数据库 | PostgreSQL + Drizzle ORM | Demo 的公共抓取缓存；长期规范化镜像数据 |
| 抓取与解析 | Node 原生 `fetch` + Cheerio | 抓取静态 JSON/HTML；不在生产抓取链路引入浏览器 |
| 确定性求解 | 独立纯 TypeScript package | 回溯/分支限界、硬约束、最小扰动和最终校验 |
| LLM 接入 | 原生 `fetch` + Zod/JSON Schema | 兼容任意 OpenAI 风格端点，避免绑定单一 SDK/供应商 |
| 向量检索（长期） | PostgreSQL `pgvector` | 评论语义检索、近重复发现和摘要取样；不作为事实源 |
| 单元/性质测试 | Vitest、fast-check | 解析器、状态机、契约和求解器性质测试 |
| 组件/E2E | Testing Library、Playwright | 关键交互、跨浏览器主流程与网络安全断言 |
| 工程质量 | ESLint、Prettier、TypeScript、GitHub Actions | lint、格式、类型、测试、构建和依赖审计 |
| 交付 | Docker 多阶段构建 + 托管 PostgreSQL | 不绑定单一云厂商；本地和线上使用同一镜像 |

**版本策略**：脚手架创建时使用各项目的稳定版并写入 `pnpm-lock.yaml`；Node 固定 24.x LTS。依赖升级通过独立 PR 完成，不在文档中写死所有 patch 版本。

## 2. 为什么适合本项目

### 2.1 模块化单体，而不是前后端两个仓库

- 当前只有一个网站、一个小团队和一条主要业务链，微服务会提前引入部署、鉴权、追踪和契约同步成本。
- Next.js App Router 能在同一项目提供 React 页面和 Route Handlers；抓取 chalaoshi 必须显式使用 **Node.js runtime**，不使用 Edge runtime。
- “单体”只描述部署单元，不代表代码混放。领域逻辑不得依赖 Next.js，未来可独立成 worker 或 API 服务。

### 2.2 PostgreSQL 从 Demo 起使用，但只存公共缓存

- D04/D31 要求 Demo 不持久化用户规划数据；因此 PostgreSQL 只保存 chalaoshi 公共缓存、抓取状态和来源元数据。
- 同一数据库可在长期阶段增加规范化教师/课程/评论表，并通过 `pgvector` 增加派生向量列，避免再维护独立向量数据库。
- Drizzle 负责类型安全查询和可审查的 SQL migration；生产环境只执行已提交 migration，不使用自动 `push` 改表。

### 2.3 IndexedDB 是用户状态的事实源

- session 和历史栈数据量、结构和查询需求均超过 `localStorage` 的合适范围；Dexie 提供 IndexedDB Schema、事务和 React live query。
- React 组件的临时状态使用内置 hooks；第一阶段**不增加 Zustand/Redux**。若后续出现跨页面瞬时状态难以维护，再用 ADR 引入。
- LLM key 与配置保存在独立 IndexedDB 表，不进入导出文件、日志或服务端持久化。IndexedDB 仍可能被同源 XSS 读取，因此 CSP、依赖审计和输出转义是必要防线，不能把“本地保存”等同于“加密保险箱”。

### 2.4 抓取采用 HTTP + HTML 解析，不运行无头浏览器

- 已知 chalaoshi 数据来自 JSON 和 HTML 页面，Node `fetch` 足以请求，Cheerio 负责静态 HTML 解析。
- 生产链路不使用 Playwright/Puppeteer 抓取：浏览器抓取镜像更大、启动慢、资源消耗高，也会让部署复杂化。
- 抓取层必须实现：来源域名白名单、超时、响应体大小限制、内容类型检查、按教师键去重、限频、3 天 TTL、失败显式化、结构校验通过后才写缓存。
- Playwright 仅用于本网站 E2E；若上游未来变为必须执行 JavaScript，先复审数据通道，不自动把浏览器抓取加入线上。

### 2.5 求解器保持纯 TypeScript

- 第一版候选集合有限，核心是可解释硬约束、志愿上限、锁定保持和最小扰动；纯函数式回溯/分支限界足够，并可用性质测试验证。
- 不引入 OR-Tools/WASM 或 Python 服务，避免双语言部署、序列化边界和原生依赖。
- LLM 只能在求解器产生的合法方案集中排序；LLM 返回结果仍由 Zod Schema 和确定性终校验拦截。

## 3. 建议仓库结构

```text
.
├── apps/
│   └── web/                    # Next.js 页面、组件、Route Handlers
├── packages/
│   ├── contracts/              # Zod Schema、DTO、错误码
│   ├── domain/                 # session、规则、志愿、状态命令
│   ├── solver/                 # 纯 TS 确定性求解器
│   ├── scraper-chalaoshi/      # HTTP client、解析器、抓取策略
│   ├── db/                     # Drizzle Schema、查询、migration
│   └── test-fixtures/          # 脱敏固定样例及 builder
├── docs/                       # 现有产品与技术文档
├── e2e/                        # Playwright 主流程与安全断言
├── infra/
│   └── docker/                 # 本地/交付所需容器配置
├── .github/workflows/          # CI
├── package.json
├── pnpm-workspace.yaml
├── pnpm-lock.yaml
└── README.md
```

依赖方向固定为：`apps/web → contracts/domain/solver/scraper/db`；`solver` 只依赖 `domain/contracts`；`domain` 不依赖 React、Next.js、数据库或抓取器。

## 4. 前后端边界

### 4.1 前端

- 交互密集页面使用 Client Components；静态外壳和无需浏览器能力的页面优先 Server Components。
- session 读写通过客户端 repository 访问 Dexie，不经网站后端。
- 用户导入文件先在浏览器校验、显示差异，再按具体功能发送最小必要字段；服务端仍需重复校验，不能信任客户端。
- LLM 能直连且 CORS 允许时由浏览器请求；需要同源代理时，key 只存在于单次请求内存，不写日志、数据库或缓存。

### 4.2 后端

- Route Handlers 只暴露版本化 REST JSON，如 `/api/v1/teachers/search`、`/api/v1/teachers/:id`、`/api/v1/llm/proxy`。
- 所有入站/出站对象使用共享 Zod Schema；错误返回稳定的 `{ code, message, requestId? }`，不得把上游 HTML 或堆栈直接返回前端。
- 求解器按 D31 在后端执行；`solver` 仍保持为不依赖 Next.js/数据库的纯 TypeScript package，便于单测、benchmark 和未来迁移。前端只发送求解所需的最小字段，后端不持久化规划数据。
- 不提供任何 zdbk 登录代理或业务写接口。

## 5. 数据设计

### 5.1 Demo 服务端表（建议最小集）

| 表 | 作用 | 关键字段 |
|---|---|---|
| `source_cache` | 公共抓取缓存 | `source_key`、`source_url`、`payload`、`content_type`、`fetched_at`、`expires_at`、`content_hash`、`parse_version` |
| `fetch_lease` | 多实例去重刷新 | `source_key`、`owner`、`lease_until` |

不创建 `user`、`session`、`plan`、`api_key` 表。过期缓存是否保留仅受 D28 的降级需求和 D03 的删除机制约束。

### 5.2 长期演进

1. 原始快照进入 S3 兼容对象存储，PostgreSQL 保存哈希、来源、抓取时间和删除状态。
2. PostgreSQL 新增规范化教师、课程、评论和来源映射表。
3. 经评测确认有收益后启用 `pgvector`；向量记录必须反查原评论和快照。
4. 向量删除由事实记录删除/停止收录事件驱动，禁止只删事实表而遗留向量。

`pgvector` 是本提案对 O-Q9 中“向量数据库”的推荐答案，但更新频率、公开镜像形态、embedding 模型和删除 SLA 仍需另行审核。

## 6. 测试与 CI 门禁

每个 PR 至少执行：

1. `pnpm lint`
2. `pnpm format:check`
3. `pnpm typecheck`
4. `pnpm test`（Vitest + fast-check）
5. `pnpm build`
6. `pnpm test:e2e`（CI 首期只跑 Chromium；发布前按 O-Q6 扩展 Firefox/WebKit）

抓取器测试只读取 `packages/test-fixtures`，普通 CI 不访问真实 chalaoshi。另设手动或定时的上游合同测试，失败只报告结构漂移，不把错误内容写入生产缓存。

GitHub Actions 使用 PostgreSQL service container 执行 migration 与缓存集成测试；CI 检查 migration 是否可从空库顺序执行。

## 7. 安全与部署基线

- 使用 Docker 多阶段构建并以非 root 用户运行；只暴露 Web 端口。
- 环境变量只包含服务端配置与数据库连接，不保存用户 LLM key。
- 默认 CSP 至少限制 `default-src`、`script-src`、`connect-src`、`frame-ancestors`；LLM 自定义端点不能通过无限放宽全站 `connect-src` 实现，优先走受控代理或明确的端点配置策略。
- 抓取 URL 由服务端根据受控 base URL 和 ID 组装，禁止接受任意 URL，避免 SSRF。
- 日志仅记录结构化错误码、耗时和匿名 request ID；key、规划内容、评论全文、Cookie/token 均需 redact。是否接入远程日志/遥测继续受 O-Q8 约束。
- 部署目标保持 Docker 兼容，可选 Railway、Render、Fly.io、云主机或其他容器平台；本轮不锁定供应商。

## 8. 未采用方案

| 方案 | 本阶段不采用的原因 |
|---|---|
| Vite SPA + Express/Fastify 两套应用 | 需要额外处理部署、同源、共享契约和两套构建；当前收益不足 |
| NestJS | 模块、装饰器和 DI 对当前小型 API 偏重；未来拆独立服务时再评估 |
| 微服务 | 团队和负载尚不需要，显著增加运维与可观测性成本 |
| MongoDB | 数据有明确关系、版本和删除一致性要求；长期还需要向量与 SQL 查询 |
| Redis 作为 Demo 缓存 | PostgreSQL 已能同时承担持久 TTL 缓存和刷新 lease，暂不增加第二个服务 |
| 独立向量数据库 | 当前没有数据规模和召回指标证明必要；`pgvector` 足够作为起点 |
| Playwright/Puppeteer 线上抓取 | 资源和部署成本过高；当前数据不需要执行 JavaScript |
| OR-Tools/Python 求解服务 | 引入双语言和原生依赖，第一版约束规模不支持这项复杂度 |
| Vercel AI SDK/单厂商 SDK 作为核心适配层 | 产品要求任意 OpenAI 兼容端点；直接维护小型协议适配器更透明 |
| Redux/Zustand 作为第一阶段全局状态 | Dexie live query + React hooks 已覆盖持久和瞬时状态；先避免双事实源 |

## 9. 审核项

请将以下五项作为一次审核结论，而不是逐个库无限讨论：

1. 是否同意 **Next.js 模块化单体 + Node.js runtime**；
2. 是否同意 **PostgreSQL 从 Demo 起只保存公共抓取缓存**；
3. 是否同意 **Dexie/IndexedDB 保存用户状态，首期不引入全局状态库**；
4. 是否同意 **纯 TypeScript 求解器 + 原生 fetch 的 OpenAI 兼容适配器**；
5. 是否同意 **Docker 可移植部署，长期优先 PostgreSQL + pgvector，而非独立向量库**。

审核通过后再执行：在 06 新增正式技术决策、更新 04/05、创建 workspace 脚手架和首批架构守护测试。
