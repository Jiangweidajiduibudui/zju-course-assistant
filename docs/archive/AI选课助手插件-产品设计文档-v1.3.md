## 1. 背景与问题

浙江大学本科选课的特点:
- **高度自由**:培养方案给出必修,但四年内修完即可毕业,不必按推荐课表;学生自主组合课程。
- **多阶段**:选课分多轮。**预置**(学校预先置入部分已选课,选课开始前完成)→ **随机选课阶段**(本工具服务的阶段)→ **补选**(申请制,非随机)。
- **顺序志愿录取**:同一时间段可为多门课/多个班填最多 **3 个顺位(志愿)**,高顺位优先;人数超容量时优先录取有优先级者(本专业/辅修),再随机。
- **选课网站**:`https://zdbk.zju.edu.cn/`(教务系统 jwglxt,选课模块 xsxk)。

**学生真实的选课流程(本产品要服务的)**:
1. 看预置课表 + 查老师评分,退课定初版;
2. 明确想上哪些课;
3. 每门课里排除"绝对不选"的班级(综合老师、给分、时间、地点);
4. 在待选班级里权衡时间/地点/评价/给分/待定人数/剩余容量,在**课程质量**与**能否抽中**之间平衡,排出最优课表。

**痛点**:第 3、4 步耗费大量精力,现有工具只帮到"信息展示",没有一个走到"决策/优化"层。

## 3. 竞品与差异化

| 工具 | 能做到 | 天花板 |
|---|---|---|
| **Lazuli** | 在 zdbk 页面内联老师 chalaoshi **评分数字** + 链接 + "待选/余量"难度比 | 只有一个评分数字;纯信息展示 |
| **Quantum(桌面)** | 登录 + 拉课程 + 心愿单 + 课表 + 快照趋势 + 一个 naive 命中概率 | chalaoshi 仍只取评分数字;**零 AI、零排序**;概率忽略"本专业优先";桌面程序需安装 |

**两者共同天花板**:① chalaoshi 只当"一个评分",没人挖**均绩 + 评价**;② 都停在"信息展示",没到"决策/优化"。

**我们的差异化(一句话)**:
> 从"信息层"跨到"决策层"——读懂 chalaoshi(评分 + **按课均绩** + **近三年评价摘要**),结合**真实命中率**与**顺位对冲**,在待选池内排出一张可执行的最优课表,并陪你交互式动态调整。

**合规立场**:对 Quantum 内网选课模块**只学思路(净室重写),不复制代码**;可复用 Lazuli(公开、宽松);chalaoshi **低频访问 + 本地缓存**,避免滥用。

## 4. 核心设计原则

1. **尽可能真实**:数据尽可能准确;无法确认时标注"缺失 / 需确认",避免呈现可能错误的信息——尤其老师评分/均绩的匹配。信任是产品命脉。
2. **全程 advise-only**:插件**永不写入选课系统**,只输出"模拟课表",由用户手动录入 zdbk。选课高风险不可逆,执行权归用户。
3. **确定性工具保正确,LLM 做模糊取舍**(harness engineering):冲突、命中率、覆盖等 correctness-critical 计算走确定性工具;偏好综合、评价摘要、解释走 LLM。外壳不把所有事塞给模型。
4. **纯 sidebar 隔离,零页面注入**:所有操作在侧边栏;对 zdbk 页面零改动(后台 AJAX + 复用会话即可)。以**单向桥接**补偿割裂成本:侧栏可一键重现 zdbk 页面当前的搜索(只读页面,不注入 UI)。
5. **YAGNI**:架构画全,只做已确认需要的部分;规则阻塞项以档位占位/降级先行(见 §15)。
6. **对话为辅但有实权且不黑箱**:LLM 的任何"改状态"动作,先转成**可见的结构化改动、由用户确认**。

## 6. 系统架构

**分层**:前端(侧边栏)/ 后端 Harness(浏览器本地)/ 引擎(LLM,用户自备,可选)/ 数据源。**只有 M4 推荐核心是真 agent,其余是确定性服务 + 视图。**

```
┌──────────────────── 外部数据源 ────────────────────┐
│  zdbk 选课系统(AJAX·复用登录态)   chalaoshi.de(评分/均绩/评价) │
└─────────────┬───────────────────────────┬──────────┘
              │ AJAX(零页面注入)          │ 按需拉取 + 本地缓存(低频)
              ▼                            ▼
┌──────────────────── 后端 · Harness(浏览器本地)────────────────────┐
│ 工具注册表                                                          │
│   读工具:getSections · getChalaoshi · getSelected(已选/预置)      │
│   算工具:checkConflicts · computeOdds🔖 · summarizeReviews(LLM)   │
│ 编排核心 M4(有界 agent loop):                                     │
│   ┌─▶ GATHER(装配上下文·确定性) → PROPOSE(LLM) → VERIFY(工具) ─┐   │
│   └──────────────── REPAIR / 结束(orchestrator 控制)──────────┘   │
│ 上下文/状态:待选池 · 规则栏(偏好) · 方案 · 缓存 · session         │
└──────────────────────────────┬─────────────────────────────────────┘
                                │ 结构化结果 + 解释
                                ▼
┌──────────────── 前端 · 侧边栏(与选课页并排)────────────────┐
│  ┌ 上半(锚):模拟课表(常驻,已选固定底座 + 规划图层[手动+推荐])┐ │
│  ├ 下半:上下文工作区(搜索结果 / 班级卡片 / 待选池 切换)   │ │
│  └ 底部:对话 + 命令栏(/new /resume …,常驻)               ┘ │
└──────────────────────────────┬──────────────────────────────┘
                                │ 确认锁定 → 填写阶段 → 检查(一致性校验)
                                ▼  advise-only(插件不写入)
                        用户手动录入 zdbk
```

**引擎**:LLM(用户自备 API key,OpenAI 兼容端点);无 key 时降级为"纯数据 + 手动"工具。

## 8. 功能模块详设

> 每个模块给出:**职责 / 输入 / 输出 / 行为与边界**。这是实现的主依据。

### M2 · 选课数据
- **职责**:全局数据源。抓取课程/教学班/已选,结构化 + 缓存。
- **输入**:类别或课名关键词;课程 id;学生会话。
- **输出**:结构化的 `Course`(代码/名/学分/学院/性质)与 `Section`(课号/教师[]/时间地点[]/考试/教学方式/**余量·容量·本专业待选·总待选**/校区)。
- **行为**:复用登录 cookie 后台 AJAX;**零页面注入**;数据只读。**单向桥接**:侧栏可读取当前 zdbk 页面的搜索上下文(关键词/分类),一键在侧栏重现同一搜索(只读页面,不注入 UI)。

### M1 · 评价(差异化的根)
- **职责**:为每个教学班的每位老师提供 chalaoshi 评分 · 按课均绩 · 点名情况 · 近三年评价摘要 · 原页链接。
- **展示**:点课程 → 该课班级**按老师归类**;每位老师一行显示 时间/地点/**评分**/**均绩**;**评价摘要不预生成**,点按钮才生成、以**展开框**呈现。
- **评价摘要内容顺序**:① 是否点名 + 点名形式 → ② 近三年评价的**优缺点如实概括**(优点缺点都要)。
- **近三年**:按评论**发布日期**取最近 3 年;摘要为"该老师总体"(评论未按课标注),**均绩按本课程**。可"重新生成"。
- **🔴 尽可能真实的匹配规则(硬要求)**:
  - 匹配键 = 姓名 +(可得的)院系;
  - **唯一命中** → 正常显示;
  - **多个同名候选** → **不猜**,标"检测到多位同名老师",让用户点选;
  - **未命中** → 明示"chalaoshi 暂无该老师数据",不留空、不瞎填;
  - **一班多师** → 每位老师**各显示一行**;
  - **均绩** → 能对上本课程才显示"该老师在〈本课程〉的均绩",对不上则显示总体并**标注"非本课程"**。
- **无 key 降级**:评分/均绩/点名/链接照常;仅"评价摘要"显示"配置 LLM 后解锁",点击直达 **key 配置向导**(§14)。

### 待选池(M3)
- **职责**:承载用户策展的候选**教学班**(section 为单位)。
- **加入**:班级卡片上 `+`;支持在一门课的班级列表**多选/全选批量入池**。**按课程分组**展示("这门课我有这 N 个候选班");支持单个或**批量移除**;去重。
- **持久化**:属于当前 session 的状态(见 §13)。

### 偏好模型 · 规则栏(rule rows)
- **职责**:让用户表达偏好,供 M4 综合。
- **结构**:一叠"规则栏"。默认预置 **1 条全局规则栏**,底部 `+` 添加。
- **每条规则栏**:**作用范围**(默认全局,可设为某门课/某类/某场景)+ **偏好顺序**(拖动排序,如 均绩>评分>时间)+ **硬约束字段**("禁止/必须",如"绝不早八""周三留空")。
- **覆盖规则**:**小范围覆盖大范围**。
- **对话补充**:LLM 可把模糊诉求("想轻松点")转成一条**可见的规则栏建议**,用户确认后生效(见"对话面板")。

### M4 · 推荐编排核心(含志愿策略)
产品的 agent 内核。采用 **Gather–Act–Verify + Reflexion** 有界循环,**orchestrator 拥有循环,不是 LLM**。
- **输入**:待选池(按课分组的候选班)+ 规则栏 + 已选/预置(固定底座)+ 手动钉/改(硬约束)。
- **循环**:
  1. **GATHER(确定性)**:装配相关小集合(候选班 + 偏好 + 当前草案 + 上轮校验反馈),**不 dump 全量**;
  2. **PROPOSE(LLM)**:基于序数偏好 + 评价文本 + 已知事实,提出/修订一版完整课表(每课选班 + 顺位),输出**结构化**;
  3. **VERIFY(确定性工具,= ground truth)**:`checkConflicts`、`computeOdds`🔖、覆盖/约束核对,返回**结构化反馈**;**绝不让 LLM 自判**;
  4. **REPAIR/结束**:有问题喂回重修;**终止条件**:校验通过且满意 / 迭代预算 / 停滞检测 / 错误预算;每轮 **re-ground**(重注入原始偏好)。
- **顺位对冲(核心机制)**:同一时间段多门课**不报错**,叠入**≤3 顺位**(高顺位优先)。当某槽候选 **>3** 时,**停下来问用户保留哪 3 个**(不擅自砍)。
- **严格池内**:推荐器**只在待选池内**决策,**绝不**引入或提示池外班级。
- **输出**:**单张**综合最优模拟课表(见 M5)。
- **🔖 依赖选课规则**:`computeOdds`(命中率)算法、"同槽顺位默认排序策略"——规则明确前以占位呈现("待规则确认"),该部分实现前必须回填。**命中率在产品内一律以档位(稳/悬/危)呈现,不展示点估计数字**——底层概率精度撑不起精确数值,档位更诚实。

### M5 · 课表可视化(锚视图,含手动规划工作台)
- **职责**:呈现方案 + 承载直接操作。**常驻侧边栏上半**,是决策锚点。
- **构成**:**已选/预置 = 固定底座(动态从正式课表同步)** + **规划图层(手动摆放与推荐结果同层,均可编辑)**。
- **交互**:
  - 从待选池**拖班级卡片到课表**摆入规划图层 = 把该班**钉为第一志愿**(对 M4 的手动硬约束);可移除、可替换为同课程其他候选班;
  - 每次摆放/变动实时跑 `checkConflicts` 并核对余量(确定性工具,**不依赖选课规则,不被 🔖 阻塞**):冲突/余量不足即时标色——用户不用生成推荐也可**纯手动摆出一张无冲突课表**(工具保正确,人做取舍);
  - 点某时间段 → 展开该槽**全部志愿** → **拖动改顺位**;
  - 每班标**命中率档位(稳/悬/危)**🔖。
- **布局(窄栏对策)**:侧栏三区(课表/工作区/对话)各自**可折叠**;课表可展开为**全 panel 视图**;IA 以 **360–420px 宽度下可读、可拖拽**为验收标准。
- **重优化**:任何手动改动后出现**"重新优化"按钮** → 点击执行**最小扰动**重排(手动改动即硬约束,其余尽量不动)+ **变更高亮**;另有**"回滚到上一状态"**逃生口(维护状态历史栈)。

### 对话 + 命令面板(常驻底部)
- **定位**:**为辅但有实权**。结构化直接操作是主干。
- **三件事**:① 把模糊诉求→**可见的结构化改动(规则栏/约束)→用户确认**;② 解释推荐(为什么这门放顺位2);③ 回答课程/老师问题。
- **session 操作**:「新规划」「历史」以**界面按钮**为主入口(目标用户是学生,不暴露命令行心智);`/new`、`/resume` 保留为等价快捷命令。
- **原则**:任何改状态动作先转结构化改动、由用户确认,**不黑箱**。

### 定稿收尾(advise-only 的最后一公里)
- **确认**按钮 → **锁定最终课表** → 进入**填写阶段**(用户去 zdbk 手动录入);
- **检查**按钮 → 重新拉 zdbk"已选"接口,把**实际选中** vs **定稿方案**逐条**一致性校验**,**指出哪里填错**(漏选/顺位反/选错班);**只提示,不代填**。

### Ma · 资料收集(later)
- 按课程聚合相关资料(智云/学在浙大/往年资源等)。仅**留接口**,不实现(later)。

### Mf · 底座
- 设置/偏好、会话复用、本地存储(`chrome.storage`)、事件、状态历史、**首次使用引导**(产品导览 + 隐私声明同意,见 §14)。

## 9. 端到端用户旅程(一次完整选课)

1. **开 session**:插件以**当下 zdbk 已填课表为基线从 0 开始**(固定底座;待选池/意图不自动继承)。
2. **搜课**:sidebar 内复用 zdbk 搜索(分类标签/搜索引擎)→ 结果列出课名+代码。
3. **看班选班**:点课程 → 班级按老师归类 → 看评分/均绩,按需生成评价摘要 → 排除绝对不选。
4. **入池**:`+` 或**多选/全选批量**把候选班加入待选池(按课分组;可单个或批量移除)。
5. **设偏好**:调整规则栏(默认1条全局,可加、可局部覆盖、可加硬约束);或对话说"想轻松点"→确认生成规则。
6. **生成推荐**:点"生成推荐" → M4 在池内排出**单张最优课表**(顺位对冲;>3 顺位时问你留哪3)。
7. **调整**:点槽拖顺位 / 拖卡片钉一志愿 / 手动改 → 点**重新优化**做最小扰动重排(变更高亮;可回滚)。对话可进一步微调。
8. **定稿**:点**确认**锁定 → 进填写阶段。
9. **手动录入 zdbk** → 回来点**检查**做一致性校验,修正填错处。
10. 下一轮:开新 session(以新基线**从 0 开始**);需要时可从历史 session 手动找回仍适用的部分(如规则栏)。

> 步骤 0(仅首次):**首次使用引导**——产品导览 + 简要隐私声明,**用户同意后方可使用**(见 §14)。

## 10. 确定性工具 vs LLM 分工

| 交给**确定性工具** | 交给 **LLM** |
|---|---|
| 时间冲突检测 | 综合序数偏好 + 评价文本做**模糊取舍** |
| 命中率计算🔖(按规则 + 余量/待选/本专业优先) | 生成推荐方案、排顺位 |
| 数据抓取与结构化、一致性校验 | 概括近三年评价、把模糊诉求转成规则栏 |
| 待选池/课表状态管理 | 用自然语言**解释**推荐 |

> 为什么不纯算法:多因素效用难精确量化,逼用户给权重=垃圾输入。为什么不纯 LLM:冲突/概率必须可信可复现,藏冲突或瞎编概率的课表=废品。

## 11. 错误与降级(尽可能真实,决不假装成功)

| 情形 | 行为 |
|---|---|
| 未登录 zdbk | 提示"请先登录 zdbk 选课系统",信息类功能待登录后可用 |
| 非选课期 / 数据为空 | 明示"当前非选课期或无可选数据";**评价查询等信息类功能仍可用** |
| chalaoshi 限流/失败 | 降级为**只显评分或标"暂无"** + 重试;不阻断其他功能 |
| chalaoshi 老师匹配不确定 | 见 M1 匹配规则(让用户确认 / 如实标缺失) |
| LLM key 无效/超时/超额 | 明确报错 + **降级到无 LLM 功能**(数据+手动仍可用) |
| 断网 | 用本地缓存 + 明示离线 |

## 12. 选课规则(设计前提)

### ✅ 已确认(可作为设计前提)
1. 选课自由:四年内修完必修即可毕业。
2. 阶段:预置(选课前完成)→ 随机选课阶段(本工具服务)→ 补选(申请制)。
3. 预置 = 已选中、**直接占用容量**;**退预置后本轮容量不释放**;无"接受"动作,只能退。
4. 录取机制 = **顺序志愿**;人数<容量全录;>容量优先录取有优先级者(本专业/辅修),再随机。
5. 系统对每班暴露:余量、容量、本专业待选、总待选(实时);上课时间为位域编码(学期+星期+单双周+节次)。
6. **选课阶段内志愿可随时修改**——"定稿 → 手动录入 → 事后检查校正"的流程因此成立,录入错误可补救。

### 🔖 待核实(进实现前必须解决,阻塞命中率与顺位)
1. **顺位档内录取细则**:某槽某志愿档申请>余量时,是否"本专业优先全录,剩余在非本专业中随机"。
2. **随机触发条件**:本专业≤余量是否必中;随机是否仅发生在"非本专业+名额不足"。
3. **志愿维度(双层?)**:是否**同时存在**"同一门课的不同教学班志愿" **和** "同一时间段不同课程的顺位"。
4. **同槽顺位默认排序策略**(工具默认按"最容易选中"排,依赖上述规则)。
5. 本专业判定口径(主修/辅修/转专业)、同轮能中几门、时间冲突是否被系统硬禁止、选课学分/门数上限、轮次结构。

> 完整核对清单见配套文件《浙大选课规则核对档》。

## 13. 状态与存储模型

- **session 制(每轮从 0 开始)**:一次规划 = 一个 session。**新 session 以当下 zdbk 已填课表为基线、从 0 开始**(待选池/意图不自动继承)。**依据**:第一轮未抽中某班,即说明其"报名人数 > 容量",下一轮几乎不可能再中——上一轮候选集对新一轮基本失效,自动继承只会引入无效候选的噪声。历史 session(其待选池/规则栏/方案)仍可随时找回,由用户按需手动取回仍适用的部分(如规则栏偏好)。
- **一个 session 的状态**:待选池、规则栏(偏好)、当前方案、对话历史、状态历史栈(供回滚)。
- **每轮/每次刷新的"现实"**:余量/待选(余量在一轮内不变;跨轮重新拉、重算命中率)、已选/预置(从正式课表同步)。
- **存储**:`chrome.storage.local`;chalaoshi 详情缓存 ~3 天;评分库离线包定期更新。

## 14. 技术约束与决策(实现指引,非详细代码)

- **形态**:Chrome 扩展 **Manifest V3**,UI 用 **Side Panel API**(侧边栏,与 zdbk 并排)。
- **数据获取**:**background service worker** 后台 AJAX 调 zdbk 接口,**复用浏览器 zdbk 登录 cookie**(host_permissions + credentials),**不注入页面**。
- **chalaoshi**:跨域请求(扩展 host 权限,不受 CORS 限制);先 `/getsession` 再取评论。
- **LLM**:**OpenAI 兼容端点**(`base URL + key + model` 三件套),用户自备;**无 key 可用**(纯数据+手动)。
- **key 配置向导(关键路径,不是 later)**:目标用户是普通本科生,"自备 key"是差异化功能(评价摘要/推荐)的实际触达率开关,不能只留一个裸设置表单。向导 = 厂商**一键预设**(DeepSeek/通义等,base URL 与推荐 model 自动填)→ **申请 key 图文教程深链** → 粘贴后**自动连通性测试**(失败给出可读原因)。"key 配置成功率"作为核心漏斗指标;免 key 路径(复用网页端 AI/订阅制)仍在 later,其优先级依此漏斗数据决定。
- **首次使用引导 + 隐私声明**:首次启动进入**产品导览**,并展示**简要隐私声明**——读取 zdbk 登录态仅用于本地后台请求;课程/偏好等数据仅在调用 LLM 时发往用户自配端点;其余数据不出本机——**用户同意后方可使用**。
- **确定性工具**(冲突/命中率/覆盖/一致性校验)可单测;LLM 部分靠样例评估 + 人工抽查。
- **合规**:不复制 Quantum 内网代码;可复用 Lazuli;chalaoshi 低频+缓存。
