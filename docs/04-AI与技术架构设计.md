# 04 · AI 与技术架构设计

- **版本**:v2.0(2026-07-08)
- **权威范围**:Chrome 扩展架构与组件职责、状态模型、确定性求解器、LLM 输入输出 Schema、Prompt 与能力检测、取消与错误处理、API key 存储、隐私/安全/提示注入、日志与可观测性。
- **总立场(D05)**:推荐核心是**确定性求解工作流**,不是 agent loop。确定性程序负责组合生成、硬约束、冲突检测、最小扰动和最终校验;LLM 只负责自然语言偏好结构化、评论理解、合法方案软排序和解释。只有 benchmark 证明 agent loop 明显更优时才重新评估(复审条件见 06 · D05)。

## 1. 扩展架构(Chrome MV3)

```
┌────────────── 外部数据源 ──────────────┐
│ zdbk(登录态 AJAX,只读白名单)  chalaoshi(低频+缓存) │
└──────────────┬───────────────┬─────────┘
               ▼               ▼
┌────────── service worker(数据与求解层)──────────┐
│ 数据客户端(zdbk / chalaoshi,契约见 03)          │
│ 确定性求解器(§3) · LLM 客户端(§4)               │
│ 状态存储(chrome.storage.local,§2)               │
└───────┬──────────────────────────┬──────────────┘
        ▼                          ▼
┌─ Side Panel(主 UI)─┐   ┌─ content script(页面增强)─┐
│ 课表锚视图 / 工作区   │   │ 读取页面搜索上下文(单向桥接)│
│ 待选池 / 规则栏 / 对话 │   │ 可注入增强 UI(D09,命名空间隔离)│
└──────────────────────┘   └──────────────────────────┘
```

### 组件职责

| 组件 | 职责 | 边界 |
|---|---|---|
| **Side Panel** | 主 UI:课表视图、工作区(搜索/班级卡片/待选池)、规则栏、对话与命令 | 纯视图 + 用户操作;不直接发外部请求 |
| **service worker** | zdbk/chalaoshi 数据获取(复用登录 cookie)、求解器、LLM 调用、状态持久化 | 只调用 [03](03-数据接口契约.md) 白名单读接口 |
| **content script** | 读取 zdbk 页面搜索上下文;可注入页面增强(D09) | **不得破坏 zdbk 原有功能**;DOM/CSS 命名空间隔离(前缀约定);随 [05](05-验收与测试计划.md) 兼容性回归;不调用任何写接口 |
| **storage** | `chrome.storage.local`:session、缓存、映射表、key | 不使用 `chrome.storage.sync`(D11) |

- **权限**:host_permissions 限定 zdbk 与 chalaoshi 域(chalaoshi 域名可配置,含备用域,见 03 §3);扩展跨域请求不受页面 CORS 限制。
- **UI 尺寸标准**:侧栏三区(课表/工作区/对话)各自可折叠;课表可展开全 panel;IA 以 **360–420px 宽度可读可操作**为验收标准。

## 2. 状态模型

### 2.1 session(D08)

- session **只由用户手动创建**;创建时以当时 zdbk 状态为**基线快照**,从零开始。
- 待选池、规则、方案、对话**不自动继承**;历史 session 只读可查,单项(如规则栏)可手动取回。
- 一个 session 的状态:`baseline`(基线快照)、`pool`(待选池)、`rules`(规则栏)、`plan`(当前方案)、`chat`(对话历史)、`history`(状态历史栈,供回滚)。

### 2.2 基线与同步(D19)

- `baseline` = { 已正式选上的教学班(固定), 已填但待筛选的志愿及其顺序(锁定) }。
- 支持**手动同步**,也可检测页面变化触发提示;同步时计算**差异集**(新增/删除/顺位变化)展示给用户;**用户确认后**才更新锁定基线,并对未锁定部分重新优化;**不自动覆盖当前方案**。

### 2.3 数据新鲜度(D20)

- 生成推荐、重新优化、最终确认前:**强制刷新**余量、待选人数、正式选课状态;UI 显示更新时间。
- 数据变化 → 重新计算;若使已锁定/定稿方案失效 → **必须提示用户**,不静默处理。

### 2.4 状态历史与回滚

- 每次"应用方案 / 重新优化 / 手动调整"入栈一个完整状态;回滚恢复到上一完整状态(PRD AC-7.3)。

## 3. 确定性求解器

### 3.1 输入 / 输出

- **输入**(D18):基线(已选=固定、已填志愿=锁定)、待选池(目标课程 + 各课候选教学班)、规则栏(D17 优先级)、手动锁定项。
- **输出**:合法方案集(供 LLM 软排序)→ 终选一份**完整推荐方案**;或"无解 + 冲突来源"。

### 3.2 流水线

```
① 输入装配:刷新数据(D20)→ 规范化候选与约束
② 硬过滤:锁定项固定;违反硬约束/与固定项时间冲突的候选剔除
③ 组合生成:按课程覆盖目标枚举/搜索可行组合(时间冲突、已核实的系统性限制)
④ 合法方案集:全部通过硬校验的组合(截断至 top-N,启发式排序控规模)
⑤ LLM 软排序(§4.3):按软偏好 + 评价文本在合法集内排序,返回首选及理由
⑥ 确定性终校验:对 LLM 选中的方案再跑一遍全部硬校验(Schema、池内性、锁定保持、冲突、覆盖)
⑦ 原子应用:校验全过 → 一次性应用并入历史栈;任何失败 → 不改动当前方案(D27)
```

- **重新优化(最小扰动)**:以当前方案为参照,锁定项不动,目标函数加入"变更数最小化"项;输出附变更集(供 UI 高亮)。
- **无解处理(D17)**:不自动放松任何规则;不允许 LLM 决定牺牲哪条硬约束;输出**冲突来源**(哪些约束互斥、涉及哪些课/班),交用户修改。
- **规则优先级(D17,固定)**:`手动锁定 > 局部硬约束 > 全局硬约束 > 局部软偏好 > 全局软偏好`。软偏好只影响 ⑤ 的排序,绝不进入硬过滤。

### 3.3 冻结部分(D01)

- **录取风险模块**:只定义接口占位(`estimateRisk(section, context) → { status: "unavailable" }`),UI 恒显"暂不可评估";算法、档位、校准在 [02](02-浙大选课规则核对档.md) 闭环 + 产品专项决策(06 · O-Q3)后再设计。
- **志愿/顺位**:方案 Schema v0 **不含顺位字段**(见 §4.3);志愿结构核实后按 [06](06-决策记录与版本变更.md) 决策流程扩展 Schema 与求解器。此为 D01 冻结的落地方式。

### 3.4 方法论定位

采用确定性工作流的依据:本任务**步骤固定、候选集合有限、硬约束可机检**——正符合 Anthropic《Building Effective Agents》的判断:"workflows offer predictability and consistency for well-defined tasks",且应"finding the simplest solution possible"。该文与 OpenAI Agents 文档仅作为**工程模式参考**,不构成"必须使用 Agent"的论据;教程型 ReAct/harness 项目与 Claude Code 镜像仓库**不作为实现依据**(后者不完整且有版权限制),仅列延伸阅读:

- Anthropic: Building Effective Agents / Effective context engineering for AI agents / Effective harnesses for long-running agents
- OpenAI: Agents SDK · Running agents
- (延伸阅读,非依据)claude-code-from-scratch、build-your-own-harness、llm-ReAct

## 4. LLM 集成

### 4.1 端点与能力检测(D10)

- 允许**任意 OpenAI 兼容端点**(base URL + key + model 三件套),用户自备。
- 配置时检测:连通性、**结构化输出能力**(JSON Schema 遵从)、超时表现、基本能力样题。
- **能力不足的端点不得开放完整推荐**(可只开放评价摘要等低结构化任务,按检测结果分级)。

### 4.2 API key 存储(D11)

- 默认长期保存于 `chrome.storage.local`;**明确提示本地明文存储风险**。
- UI 仅显示掩码;**不进入日志**;不使用 `chrome.storage.sync`;提供替换与一键删除。
- key **永不进入提示词**(D04)。

### 4.3 LLM 任务与 I/O Schema(D25)

四类任务,全部**固定 JSON Schema 输出**;Schema 校验失败即任务失败,**不从自然语言猜测结果**;LLM 输出永不直接写入 zdbk。

| 任务 | 输入(要点) | 输出 Schema(要点) |
|---|---|---|
| ① 偏好结构化 | 用户自然语言 + 现有规则栏 | `{ suggestedRules: [{scope, ordering[], hardConstraints[]}], explanation }` → 渲染为规则栏建议,**用户确认后生效** |
| ② 评论摘要 | 近五年评论(筛选/去重/限长后,标注为不可信数据)+ 教师/课程上下文 | `{ attendance: {called, form}, pros[], cons[], sampleSize, lowSample }` → 只读展示,**不得触达状态修改**(D24) |
| ③ 方案软排序 | 合法方案集(编号)+ 规则栏 + 相关评论要点 | `{ chosenPlanId, ranking[], reasons[] }` → `chosenPlanId` 必须属于输入集合,否则校验失败 |
| ④ 解释 | 当前方案 + 求解过程摘要 | `{ answer }`(纯文本回答,无状态操作) |

**校验链(⑥ 终校验的一部分)**:Schema 合法 → 教学班 ID 全部存在且**池内** → 课程覆盖满足 → 锁定状态保持 → 硬约束通过 → 才能展示(D25)。

### 4.4 生成过程与并发(D27)

- 展示阶段进度(装配→求解→排序→校验);**允许取消**。
- 取消/失败/校验不通过 → 不修改当前方案;完整成功 → 一次性原子应用。
- **请求代际标记**:每次生成携带递增代际号,旧请求返回时若代际落后于用户最新操作则丢弃(旧结果不能覆盖新操作)。

### 4.5 调用预算(D26)

- 产品**不限制** LLM 调用次数、超时上限、token 或费用——由用户配置的端点自行控制。
- 设置页提醒:费用风险与长时间等待风险。

### 4.6 降级(D21)

- 无 key 或 LLM 不可用:**不生成推荐**;评价摘要、偏好理解、解释禁用;浏览/评分/均绩/点名/待选池照常。UI 状态见 PRD FR-12。

## 5. 隐私与安全

### 5.1 数据发送范围(D04)

- 首次引导**一次性征得同意**(D23);同意后调用 LLM 时可发送:**完整课表、偏好、对话**。
- **永不进入提示词**:姓名、学号、Cookie、会话令牌、API key。实现要求:提示词装配处设置**出站过滤器**(字段白名单装配,而非黑名单剔除)。
- 隐私声明必须说明:**第三方模型供应商可能保存请求数据**;其余数据不出本机。

### 5.2 提示注入(D24)

- 评论文本是不可信输入。当前决定:**仅通过系统提示词声明**评论为不可信数据,不建专门隔离层——**接受此防护较弱的风险**(复审条件见 06 · D24)。
- 架构兜底:评论摘要任务(§4.3-②)**无任何状态修改通道**——其输出 Schema 为纯展示数据,且该任务的执行路径不接入规则/池/方案写操作。

### 5.3 同意与清除(D23)

- 不同意隐私声明则无法进入产品;设置页提供隐私说明与**本地数据清除**入口(session、缓存、映射、key 全清)。

## 6. 日志与可观测性

- **待产品负责人决定**(06 · O-Q8):是否设置日志、诊断导出与用户反馈机制。
- 已定硬约束:无论方案如何,**key 与 cookie 不进日志**(D11);默认不外发任何遥测。
