# 04 · AI 与技术架构设计

- **版本**:v2.4(2026-07-09)
- **权威范围**:Web 前后端架构与组件职责、chalaoshi 抓取/缓存/镜像与向量索引、状态模型、确定性求解器、LLM 输入输出 Schema、API key 存储、隐私/安全/提示注入、日志与可观测性。
- **总立场(D05、D36、D39)**:推荐核心是**集中选课数学模型 + 两阶段 LLM 排序**，不是 agent loop。`selection-model` 负责组合生成、硬约束、冲突检测、最小扰动和最终校验；LLM 负责自然语言偏好结构化、评论理解、组内软排序、Top10 完整方案比较和解释。只有 benchmark 证明 agent loop 明显更优时才重新评估(复审条件见 06 · D05)。

## 1. Web 网站架构(D31、D32、D33)

```text
┌──────────── React + Vite Web 前端 ────────────┐
│ 课程搜索 / 教师卡 / 待选池 / 规则 / 课表 / 对话  │
│ session、映射表、端点配置与用户本地状态             │
└───────────────────┬───────────────────────┘
                    │ HTTPS / JSON
                    ▼
┌────────────── Fastify 网站后端 API ──────────────┐
│ 导入校验 / chalaoshi 抓取与解析 / 确定性求解器    │
│ 请求去重、限频、缓存 / 强制同源 LLM 代理             │
└────────┬─────────────────┬──────────────────┘
         │ Demo            │ 长期
         ▼                 ▼
┌─ chalaoshi 公开资源 ─┐  ┌──────── 镜像数据层 ────────┐
│ 索引 / 详情 / 评论   │  │ 原始快照 / 规范化库 / 向量索引 │
└──────────────────────┘  └─────────────────────────────┘
```

### 组件职责

| 组件 | 职责 | 边界 |
|---|---|---|
| **React/Vite 前端** | 课程浏览、教师卡、待选池、规则栏、课表、方案和对话 | 不读取 zdbk DOM/Cookie，不直接抓取 chalaoshi |
| **Fastify 后端 API** | 导入数据校验、chalaoshi 抓取/解析、求解、缓存和强制同源 LLM 代理 | 无 zdbk 写接口能力；不持久化用户 zdbk 密码/Cookie/token 或 LLM key |
| **Dexie/IndexedDB** | session、待选池、规则、方案、映射表、用户端点配置 | 用户持久状态的唯一事实源；提供全量清除；key 后端不保存、不记录 |
| **L1/L2 后端缓存** | 进程内 LRU + PostgreSQL 保存 chalaoshi 公共缓存 | 按来源和教师 ID 键控制 TTL；不与用户身份绑定；不增加 Redis |
| **镜像/向量数据层（长期）** | 原始快照、规范化查询和评论语义索引 | 向量索引是派生层；每条数据可追溯、可过期、可删除 |

- **抓取边界**:chalaoshi 跨域请求只由后端发起；前端不依赖对方 CORS 配置。
- **zdbk 边界**:Demo 只处理手动输入/导入或脱敏 fixture；在确定官方授权、本地辅助程序或其他合法通道前，后端不代理登录态 zdbk 请求。
- **UI 形态**:不再受侧边栏宽度限制；浏览器兼容范围、响应式断点和无障碍指标仍由 06 · O-Q6 定标。
- **实现基线**:单 package、ESM、TypeScript strict；目录与依赖方向、精确版本和易错 API 以 [07](07-技术栈与版本基线.md) 为准。

## 2. 状态模型

### 2.1 session(D08)

- session **只由用户手动创建**;创建时以用户输入/导入或 Demo fixture 中的当前状态为**基线快照**,从零开始。
- 待选池、规则、方案、对话**不自动继承**;历史 session 只读可查,单项(如规则栏)可手动取回。
- 一个 session 的状态:`baseline`(基线快照)、`pool`(待选池)、`rules`(规则栏)、`plan`(当前方案)、`chat`(对话历史)、`history`(状态历史栈,供回滚)。

### 2.2 基线与同步(D19)

- `baseline` = { 已正式选上的教学班(固定), 已填但待筛选的志愿及其顺序(锁定) }。
- 支持用户重新输入/导入后的**手动同步**;同步时计算**差异集**(新增/删除/顺位变化)展示给用户;**用户确认后**才更新锁定基线,并对未锁定部分重新优化;**不自动覆盖当前方案**。

### 2.3 数据新鲜度(D20)

- 生成推荐、重新优化、最终确认前:刷新后端可获取的 chalaoshi/公开数据;对无法自动刷新的 zdbk 导入数据显示导入时间并要求用户确认。
- 数据变化 → 重新计算;若使已锁定/定稿方案失效 → **必须提示用户**,不静默处理。

### 2.4 状态历史与回滚

- 每次"应用方案 / 重新优化 / 手动调整"入栈一个完整状态;回滚恢复到上一完整状态(PRD AC-7.3)。

### 2.5 志愿模型(D30)

- 仅用于第一、二轮概率筛选;第三/四轮、补选和低分定点选课不进入模型。
- 每个志愿项必须引用一个具体教学班 `sectionId`，顺位为 `1 | 2 | 3`。
- 同时维护两类约束:同一课程最多 3 个志愿;同一时间段最多 3 个志愿。两类校验必须同时通过。
- 同一课程 2 个及以上教学班形成课程志愿组；课程志愿组优先于时间槽志愿组，冲突时时间槽志愿组失效(D37)。
- 时间槽只区分春/夏/秋/冬学期、星期几、第几节，不区分单双周。
- 上课时间重叠不直接判无解；不同课程考试时间重叠为硬无解；考试时间缺失的教学班停留在待选池。
- 课程优先级只依据课程面向对象;第一版不建模额外次级优先顺序，不使用学分作为筛选因素。
- 用户方案必须填写学分上限；教学班学分缺失时停留在待选池，不参与排课(D38)。
- 待选人数包含三个志愿档;模型对各档使用同一聚合口径，必须标记为简化估计。

## 3. 集中选课数学模型

### 3.1 输入 / 输出

- **输入**(D18、D30):基线(已选=固定、已填志愿=锁定)、待选池(目标课程 + 各课候选教学班)、课程/时间段志愿组、课程面向对象、规则栏(D17 优先级)、手动锁定项。
- **输出**:合法方案集(供 LLM 软排序)→ 终选一份**完整推荐方案**，其中志愿建议引用具体教学班并同时满足两类三志愿上限;或"无解 + 冲突来源"。

### 3.2 流水线

```
① 输入装配:刷新数据(D20)→ 规范化候选与约束
② 硬过滤:锁定项固定;违反硬约束/与固定项时间冲突的候选剔除
③ LLM 组内排序(§4.3):对每个志愿组内教学班按软偏好、评价文本和缺失中性原则排序
④ 组合生成:按 LLM 组内顺序枚举/搜索可行组合，输出 Top10 完整候选方案
⑤ LLM 方案比较(§4.3):在 Top10 完整候选中返回首选及理由
⑥ 确定性终校验:对 LLM 选中的方案再跑一遍全部硬校验(Schema、池内性、锁定保持、考试冲突、学分、覆盖)
⑦ 原子应用:校验全过 → 一次性应用并入历史栈;任何失败 → 不改动当前方案(D27)
```

- **重新优化(最小扰动)**:以当前方案为参照,锁定项不动,目标函数加入"变更数最小化"项;输出附变更集(供 UI 高亮)。
- **无解处理(D17)**:不自动放松任何规则;不允许 LLM 决定牺牲哪条硬约束;输出**冲突来源**(哪些约束互斥、涉及哪些课/班),交用户修改。
- **规则优先级(D17,固定)**:`手动锁定 > 局部硬约束 > 全局硬约束 > 局部软偏好 > 全局软偏好`。软偏好只影响 ⑤ 的排序,绝不进入硬过滤。

### 3.3 规则解冻与保留冻结(D01、D30)

- **志愿/顺位已解冻**:按 §2.5 建模，可生成结构化志愿建议，但仍只由用户手动录入 zdbk。
- **录取风险数值仍冻结**:待选人数为三档聚合值，缺少真实顺位分布与校准数据;接口保持 `estimateRisk(section, context) → { status: "unavailable" }`，UI 恒显"暂不可评估"。

### 3.4 方法论定位

采用确定性工作流的依据:本任务**步骤固定、候选集合有限、硬约束可机检**——正符合 Anthropic《Building Effective Agents》的判断:"workflows offer predictability and consistency for well-defined tasks",且应"finding the simplest solution possible"。该文与 OpenAI Agents 文档仅作为**工程模式参考**,不构成"必须使用 Agent"的论据;教程型 ReAct/harness 项目与 Claude Code 镜像仓库**不作为实现依据**(后者不完整且有版权限制),仅列延伸阅读:

- Anthropic: Building Effective Agents / Effective context engineering for AI agents / Effective harnesses for long-running agents
- OpenAI: Agents SDK · Running agents
- (延伸阅读,非依据)claude-code-from-scratch、build-your-own-harness、llm-ReAct

## 4. LLM 集成

### 4.1 端点与能力检测(D10)

- 允许**任意 OpenAI 兼容端点**(base URL + key + model 三件套),用户自备。
- 配置时检测:连通性、**结构化输出能力**(JSON Schema 遵从)、超时表现、基本能力样题。
- **能力不足的端点不得开放完整推荐**(可只开放评价摘要等低结构化任务,按检测结果分级)。

### 4.2 API key 与代理(D11、D40)

- 所有 LLM 请求必须通过同源 Fastify 后端代理；前端不直连模型端点。
- key 只随单次请求进入后端内存，**不落库、不写日志、不进入提示词**。
- 若前端选择本地保存 key，必须明确提示本地明文存储风险，并提供替换与一键删除。
- 代理仅允许 HTTPS，阻断 localhost/private/link-local/reserved IP，DNS 与重定向后都要复核，并限制端口、请求体、超时、并发和响应大小。

### 4.3 LLM 任务与 I/O Schema(D25)

四类任务,全部**固定 JSON Schema 输出**;Schema 校验失败即任务失败,**不从自然语言猜测结果**;LLM 输出永不直接写入 zdbk。

| 任务 | 输入(要点) | 输出 Schema(要点) |
|---|---|---|
| ① 偏好结构化 | 用户自然语言 + 现有规则栏 | `{ suggestedRules: [{scope, ordering[], hardConstraints[]}], explanation }` → 渲染为规则栏建议,**用户确认后生效** |
| ② 评论摘要 | 近五年评论(筛选/去重/限长后,标注为不可信数据)+ 教师/课程上下文 | `{ attendance: {called, form}, pros[], cons[], sampleSize, lowSample }` → 只读展示,**不得触达状态修改**(D24) |
| ③ 组内排序 | 志愿组内教学班(编号)+ 用户偏好 + 相关评论要点 | `{ groupRankings: [{groupId, orderedSectionIds[], reasons[]}] }` → ID 必须属于输入组，缺失软指标保持中性 |
| ④ 方案比较 | Top10 完整候选方案(编号)+ 规则栏 + 相关评论要点 | `{ chosenPlanId, ranking[], reasons[] }` → `chosenPlanId` 必须属于输入集合,否则校验失败 |
| ④ 解释 | 当前方案 + 求解过程摘要 | `{ answer }`(纯文本回答,无状态操作) |

**校验链(⑥ 终校验的一部分)**:Schema 合法 → 教学班 ID 全部存在且**池内** → 课程覆盖满足 → 课程/时间段志愿组均不超过 3 个且顺位唯一 → 锁定状态保持 → 硬约束通过 → 才能展示(D25、D30)。

### 4.4 生成过程与并发(D27)

- 展示阶段进度(装配→求解→排序→校验);**允许取消**。
- 取消/失败/校验不通过 → 不修改当前方案;完整成功 → 一次性原子应用。
- **请求代际标记**:每次生成携带递增代际号,旧请求返回时若代际落后于用户最新操作则丢弃(旧结果不能覆盖新操作)。

### 4.5 调用预算(D26)

- 产品**不限制** LLM 调用次数、超时上限、token 或费用——由用户配置的端点自行控制。
- 设置页提醒:费用风险与长时间等待风险。

### 4.6 降级(D21)

- 无 key 或 LLM 不可用:**不生成推荐**;评价摘要、偏好理解、解释禁用;浏览/评分/均绩/点名/待选池照常。UI 状态见 PRD FR-12。

## 5. 隐私与安全

### 5.1 数据发送范围(D04)

- 首次引导**一次性征得同意**(D23);网站必须分别说明课程/基线数据提交给后端处理、chalaoshi 后端抓取，以及调用 LLM 时可能发送的**完整课表、偏好、对话**。
- **永不进入提示词**:姓名、学号、Cookie、会话令牌、API key。实现要求:提示词装配处设置**出站过滤器**(字段白名单装配,而非黑名单剔除)。
- 隐私声明必须说明:**网站后端会处理用户主动提交的规划数据，第三方模型供应商可能保存请求数据**;Demo 不持久化用户规划数据，除非未来另行增加账号与云同步决策。

### 5.2 提示注入(D24)

- 评论文本是不可信输入。当前决定:**仅通过系统提示词声明**评论为不可信数据,不建专门隔离层——**接受此防护较弱的风险**(复审条件见 06 · D24)。
- 架构兜底:评论摘要任务(§4.3-②)**无任何状态修改通道**——其输出 Schema 为纯展示数据,且该任务的执行路径不接入规则/池/方案写操作。

### 5.3 同意与清除(D23)

- 不同意隐私声明则无法进入产品;设置页提供隐私说明与**客户端数据清除**入口(session、映射、key 全清)。后端 chalaoshi 公共缓存不与用户身份关联。

## 6. 日志与可观测性(D42)

- 日志采用稳定 JSON Lines，包含 `requestId`、`generationId`、`module`、`action`、`durationMs`、`status`、`errorCode`、`schemaVersion` 等字段，便于 AI 维护。
- 默认不外发任何遥测；诊断导出由用户主动触发。
- 禁止记录课程详情全文、用户偏好全文、评论原文、LLM prompt/response、key、zdbk 密码、Cookie、会话令牌、学号姓名。

## 7. 工程质量基线

- TypeScript 独立实现(D02);确定性求解器与解析器可单测;LLM 部分用固定评测集(见 [05](05-验收与测试计划.md))。
- 正式技术栈为 React + Vite + Fastify + PostgreSQL；所有直接依赖精确锁定，第三方 API 修改前强制查询 Context7 对应版本文档(D33、D34；见 [07](07-技术栈与版本基线.md))。
- 不复制 Lazuli(GPLv3)与校内 Quantum 代码;参考公开 Quantum(MIT)架构时保留许可证声明。
