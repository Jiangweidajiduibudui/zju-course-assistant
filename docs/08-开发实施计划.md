## 1. 一句话目标

由负责人带领 5 名组员，用一套 TypeScript 模块化单体完成一个**能直接正式演示**的浙大 AI 选课助手网站：用户导入课程数据、配置学分上限和偏好，系统抓取或读取 chalaoshi 数据，通过集中选课数学模型与两阶段 LLM 排序生成可解释、可手动录入 zdbk 的志愿方案与预期课表。

## 2. 本阶段不做什么

1. 不创建 Chrome 插件，不注入 zdbk 页面，不读取浏览器 zdbk Cookie。
2. 不自动登录 zdbk，不调用选课、退选、调序等业务写接口。
3. 不做第三/四轮、补选、低分定点、退课和放弃修读。
4. 不展示精确中签概率、“稳/悬/危”档位或未校准风险分。
5. 不提交真实 chalaoshi 评论到公开 GitHub。
6. 不引入微服务、动态插件框架、第二种后端语言、Redis 或独立向量数据库。
7. 不用 mock 结果伪装正式推荐；测试夹具只能用于契约测试和演示兜底说明。

## 3. 正式 Demo 定义

正式 Demo 必须一次跑通以下闭环：

1. 用户完成首次隐私同意。
2. 加载内置合成 Demo 数据，或导入指定 JSON。
3. 用户填写必填学分上限。
4. 用户浏览课程、教学班、教师评价状态和缺失字段说明。
5. 用户把具体教学班加入待选池。
6. 系统按课程志愿组与时间槽志愿组生成“待筛选志愿”视图。
7. chalaoshi 后端抓取成功时展示真实来源与时间；失败时使用合成 seed cache 并明确标记。
8. 用户配置 OpenAI 兼容端点；所有 LLM 请求通过同源 Fastify 代理。
9. LLM 先对每个志愿组内教学班进行模糊排序。
10. `selection-model` 生成 Top10 合法候选方案。
11. LLM 比较完整候选方案并返回首选与理由。
12. `selection-model` 对 LLM 结果做最终硬校验。
13. UI 展示“志愿提交方案”和“预期课表投影”。
14. 用户可手动调整、锁定、重新优化、回滚。
15. 用户导出当前 JSON，用于复查或后续导入。
16. E2E 证明全程没有 zdbk 写请求。

### 3.1 Demo 数据策略

- 仓库允许提交结构忠实的合成 seed cache，教师、评论和课程细节不对应真实个人评价。
- live chalaoshi 抓取成功时优先使用真实实时数据，并显示来源链接、抓取时间、缓存状态。
- 抓取失败、网络受限或缓存为空时，Demo 可降级到合成 seed cache；UI 必须显式标记“演示数据”。
- CI 默认只跑合成 fixture，不访问真实 zdbk 或 chalaoshi。

### 3.2 与“先跑通主干”的关系

本项目不设置对外的假 Demo 或 mock 版 walking skeleton。团队内部仍按持续集成推进：

- 每个模块先交契约、fixture 和测试，再接入真实页面。
- 未完成页面隐藏在正式导航外。
- 主分支始终保持类型检查、测试、构建通过。
- 模块未完成时可以使用 fixture 证明契约，但不能在正式入口伪装成真实功能。

## 4. 总体架构

```text
React + Vite Web
  ├─ 待筛选志愿页：课程志愿组 / 时间槽志愿组 / 顺位调整 / 缺失字段解释
  ├─ 预期课表页：首选投影 / 备选堆叠 / 冲突与未知标记
  ├─ 设置与导入：隐私同意 / LLM 端点 / 学分上限 / JSON 导入导出
  └─ 本地状态：Dexie session / 待选池 / 锁定 / 回滚 / 诊断导出
          │
          ▼ same-origin JSON
Fastify API
  ├─ import：JSON 校验、规范化、错误定位
  ├─ chalaoshi：限频抓取、解析、L1/L2 缓存、合成 seed fallback
  ├─ llm-gateway：OpenAI 兼容代理、SSRF 防护、结构化输出校验
  ├─ planner：编排 LLM 排序、selection-model、最终校验
  └─ diagnostics：结构化日志与本地诊断导出
          │
          ▼ pure TypeScript domain
selection-model
  ├─ 时间槽归一化：学期 + 星期 + 节次，不区分单双周
  ├─ 志愿组构造：课程组优先，冲突时时间槽组失效
  ├─ 可排性判断：考试硬冲突、学分上限、缺失硬字段
  ├─ 候选枚举：按 LLM 组内顺序生成 Top10 完整方案
  ├─ 投影：志愿提交方案 → 预期课表
  └─ 终校验：LLM 输出、锁定项、硬约束、可解释错误
```

## 6. 选课数学模型的最终口径

### 6.1 集中模块原则

所有影响推荐正确性的规则都进入 `src/domain/selection-model/`：

- 课程志愿组与时间槽志愿组构造。
- 课程组优先导致时间槽组失效的判定。
- 春/夏/秋/冬 + 星期 + 节次的时间槽归一化。
- 考试时间重叠硬校验。
- 考试时间缺失、学分缺失导致候选停留在待选池。
- 用户必填学分上限校验。
- Top10 完整候选方案生成。
- 志愿提交方案与预期课表投影。
- 手动锁定、重新优化、最小扰动、回滚前校验。
- LLM 输出最终硬校验。

前端只渲染状态，后端只编排流程；任何地方不得重复实现一份“简化版规则”。

### 6.2 志愿组规则

1. 同一课程出现 2 个及以上候选教学班时形成课程志愿组。
2. 课程志愿组最多 3 个具体教学班，顺位为 1/2/3。
3. 被课程志愿组占用的教学班不能再进入时间槽志愿组。
4. 若某个时间槽组与课程志愿组冲突，优先服从课程志愿组，时间槽组失效。
5. 课程组缩小到 1 个教学班时，课程组自动消解；该教学班可以参与时间槽组。
6. 时间槽志愿组最多 3 个具体教学班，顺位为 1/2/3。

### 6.3 时间与考试规则

| 项 | 规则 |
|---|---|
| 上课时间粒度 | 春/夏/秋/冬学期 + 星期几 + 第几节；不区分单双周。 |
| 上课时间重叠 | 不直接判无解；进入组内排序和 LLM 判断，由最终方案投影解释。 |
| 考试时间重叠 | 不同课程在同一时间段考试重叠，判硬无解。 |
| 同课考试重叠 | 同一课程的不同教学班考试时间重叠属于正常情况。 |
| 考试时间缺失 | 教学班标记为 `unknownExamTime`，留在待选池，不参与排课。 |

### 6.4 学分规则

- 每个 session 必须填写用户自己的学分上限。
- 首版不设置课程门数上限。
- 教学班学分缺失时标记为 `unknownCredit`，留在待选池，不参与排课。
- 学分不进入录取优先级或概率估计；只作为用户方案的总量硬约束。

## 7. AI 与 LLM 工作流

### 7.1 两阶段 AI

```text
输入待选池 + 用户偏好 + chalaoshi 摘要
  → LLM 按志愿组内教学班排序
  → selection-model 按排序枚举 Top10 完整候选方案
  → LLM 比较 Top10 完整候选并给理由
  → selection-model 终校验
  → UI 展示方案或错误
```

设计原因：本项目引入 AI 的价值在于处理难量化的软偏好，例如教师评价、评论语义、时间舒适度、用户自然语言目标。硬约束、组合合法性和最终状态必须由可测试的确定性模块保证。

### 7.2 缺失数据处理

| 数据类型 | 缺失时处理 |
|---|---|
| 教师 GPA、评分、评论、点名信息 | 软指标缺失保持中性，交给 LLM 综合判断。 |
| 考试时间 | 标记未知，留在待选池，不参与排课。 |
| 学分 | 标记未知，留在待选池，不参与排课。 |
| 容量、待选人数 | 不生成风险概率；可以展示字段缺失或“暂不可评估”。 |

### 7.3 LLM 代理安全

- 前端不直连模型端点，所有请求走同源 Fastify。
- key 只随单次请求进入后端内存，不落库、不写日志、不进入 prompt。
- 仅允许 HTTPS。
- 阻断 localhost、private、link-local、reserved IP。
- DNS 解析后检查 IP，重定向后再次检查。
- 限制端口、请求体大小、超时、并发和响应大小。
- LAN 模型端点默认不可用，除非未来另行加入本地部署安全模式。

## 8. UI 实现口径

### 8.1 待筛选志愿页

参考 zdbk 选课网站的待筛选志愿心智模型：

- 以课程志愿组和时间槽志愿组分块展示。
- 每个志愿项显示课程、教学班、教师、时间、考试、学分、chalaoshi 状态、缺失字段。
- 支持上移/下移调整顺位，不做自由拖拽。
- 用户手动调整后，该组顺位被锁定；重新优化不得改动锁定顺位。
- 被课程志愿组占用而导致时间槽组失效的项必须说明原因。

### 8.2 预期课表页

参考 zdbk 预期课表心智模型：

- 课表只投影当前首选方案。
- 同一格可以标记备选堆叠，但不得表现成用户会同时上多门互斥课程。
- 考试硬冲突直接在方案生成前阻断。
- 考试时间未知、学分未知的教学班显示在待选池说明区，不进入课表。
- 支持查看“为什么这样排”：显示 LLM 软理由和 deterministic 校验摘要。

## 10. 开发顺序与门禁

### Task 0 · 契约与合成 fixture

- 负责人建立 `shared/contracts` 的命名、错误码、日志字段和 fixture 目录。
- A/B/C/D/E 基于同一份合成 fixture 开发。
- 门禁：契约测试能证明 fixture 可被解析，且没有真实评论、学号、姓名、Cookie、key。

### Task 1 · selection-model 先行

- C 先交纯函数模型和测试，不依赖 React、Fastify、LLM。
- 覆盖课程组优先、时间槽失效、考试硬冲突、考试未知、学分未知、学分上限、Top10、投影。
- 门禁：所有冲突样例都有稳定错误码；性质测试覆盖“终校验永不接受非法方案”。

### Task 2 · 导入、池与本地状态

- A 交 JSON 导入、字段修正、导出；E 接 Dexie session 和待选池 UI。
- 门禁：导入 → 修改 → 导出 → 再导入结果一致；缺失硬字段正确留在待选池。

### Task 3 · chalaoshi 与合成 seed fallback

- B 交抓取、解析、缓存、失败降级和来源状态。
- 门禁：网络失败时 UI 不崩溃；合成 seed 被清晰标记为演示数据；真实评论不进入仓库。

### Task 4 · LLM 网关与两阶段 AI

- D 交同源代理、安全检查、结构化输出、组内排序和 Top10 方案比较。
- C 提供终校验接口，负责人接入 planner。
- 门禁：无 key 不生成推荐；SSRF 样例被拒；LLM 返回不存在 ID 时不改状态。

### Task 5 · zdbk-like 工作台

- E 交待筛选志愿页、预期课表页、锁定、重新优化、回滚和诊断导出。
- 门禁：手动锁定后重新优化不改变锁定组；课程组优先导致时间槽失效的解释可见。

### Task 6 · 正式 Demo 集成

- 负责人打通从导入到导出的全链路。
- 门禁：Playwright 跑通正式 Demo；`verify-no-zdbk-write` 通过；日志导出不含敏感字段。

### Task 7 · GitHub 同步前硬化

- 清理 `.gitignore`、README、文档地图、合成 fixture、许可证状态说明。
- 门禁：`git diff --check`、格式化、类型检查、测试、构建、E2E、敏感字段扫描全部通过。

## 12. AI 友好日志规范

日志以 JSON Lines 输出，每行是一条稳定 Schema 的事件：

```json
{
  "ts": "2026-07-09T10:00:00.000+08:00",
  "level": "info",
  "requestId": "req_...",
  "generationId": "gen_...",
  "module": "selection-model",
  "action": "final_validate",
  "status": "ok",
  "durationMs": 12,
  "errorCode": null,
  "schemaVersion": "log.v1"
}
```

禁止记录：课程详情全文、用户偏好全文、评论原文、LLM prompt/response、API key、Cookie、zdbk token、姓名、学号、身份证、手机号。诊断导出由用户主动触发，默认不外发远程遥测。
