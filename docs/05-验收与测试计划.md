## 3. 求解器正确性测试

### 3.1 性质测试(property tests)

对随机生成的候选/约束输入,任何输出方案必须满足:

1. **考试无硬冲突**:不同课程不得出现同一考试时间;同一课程不同教学班考试重叠允许——D37;
2. **池内性**:所有推荐教学班 ∈ 待选池——AC-4.2;
3. **锁定保持**:已选固定、已填志愿锁定、手动锁定不变——AC-6.1/6.2/7.1;
4. **硬约束满足**:必填学分上限、考试时间、锁定项和志愿组规则均通过;无解时给出冲突来源且不放松规则——AC-5.2;
5. **最小扰动**:重新优化的变更集不含锁定项,且不存在"明显更少变更"的等效解(以启发式上界断言);
6. **原子性**:取消/失败路径不留下半成品状态——AC-6.4。
7. **志愿合法性**:任一志愿都指向池内具体教学班，同课程和同时间段的志愿均不超过 3 个，两类约束同时成立——D30。
8. **模型边界**:第三/四轮、补选、学分因素不得进入录取优先级或概率估计；学分只作为用户方案硬上限——D30、D38。
9. **缺失硬字段**:考试时间或学分缺失的教学班不得进入排课结果，只能留在待选池并给出原因——D37、D38。
10. **Top10 边界**:候选方案输出最多 10 个，且每个候选都能独立通过终校验——D39。

### 3.2 虚拟模型 benchmark(离线仿真)

构建模拟课程/容量/待选人数的虚拟数据集,离线检验:冲突检测、覆盖求解、最小扰动的正确性与性能上限(候选规模压力测试)。

> ⚠️ 边界:benchmark 可按 D30 模拟“课程面向对象 → 志愿顺序 → 同档随机”的结构正确性，但待选人数仍是三档聚合值。在获得真实顺位分布与校准数据前，**不得将仿真输出解释为真实录取概率**(06 · O-Q3)。

## 4. LLM 固定评测集

- **偏好结构化**:固定语料("想轻松点""周三留空"等)→ 期望规则栏结构;评 Schema 合规率与语义正确率(人工评分)。
- **评论摘要**:固定评论集 → 必须同时含 pros 与 cons(AC-3.4);低样本标记正确(AC-3.3);含提示注入样本(评论中夹带指令)→ 摘要输出不得包含指令执行迹象,状态零变更(D24)。
- **组内排序**:固定志愿组 → `orderedSectionIds` 必须只包含输入组内 ID；缺失 GPA/评分/评论时不得固定扣分，按中性信息处理(D39)。
- **方案比较**:固定 Top10 合法方案集 → `chosenPlanId` 必须在集合内;Schema 失败率统计(D25、D39);排序与偏好一致性抽查。
- **能力检测样题**(D10):作为向导检测的固定题库,含结构化输出与超时用例。

## 5. 端到端与回归

### 5.1 Web E2E(导入 fixture + mock chalaoshi)

- 主流程:首次同意(AC-11.1/11.2)→ 新建 session(AC-1.1)→ 填写学分上限(D38)→ 导入/搜课(AC-2.1/2.2)→ 评价卡(AC-3.x)→ 入池(AC-4.1)→ 待筛选志愿页(D42)→ 生成 Top10 并选择(AC-6.x/D39)→ 预期课表页(D42)→ 调整/回滚(AC-7.x)→ 导出 JSON/定稿检查(AC-8.1)。
- 降级矩阵:未导入课程/基线 / 导入非法 / chalaoshi 抓取不可达、限流或超时 / 无 key / 能力检测不通过 / Schema 失败 / 断网(AC-12.1/12.2;AC-10.1)。
- 对话确认制:未确认不改状态(AC-9.1)。
- key 生命周期:配置/掩码/替换/删除(AC-10.2/10.3);服务端代理不保存、不记录 key(D40)。
- SSRF 防护:localhost、私网、link-local、reserved IP、危险重定向和超限 body 必须被拒绝(D40)。
- 日志/诊断导出:JSON Lines 可被 AI 读取，且不包含课程详情全文、偏好全文、评论原文、LLM I/O、key、Cookie、学号姓名(D42)。
- **zdbk 零调用**:Demo E2E 全程网络记录中不得出现任何 zdbk 请求，也不得上传 zdbk 密码、Cookie 或会话令牌(AC-2.3/8.2)。

### 5.2 网站与抓取回归(D31、D32)

- 主流桌面浏览器下导入、搜索、待选池、课表、推荐和对话主流程可用;具体浏览器矩阵待 O-Q6 定标。
- chalaoshi 上游结构改变时解析器必须显式失败，不用部分错位数据更新缓存。
- 对相同教师的并发请求只触发一次上游抓取;限频和 TTL 行为可确定验证。

### 5.3 CI 必过项(D34)

每个合并请求按顺序执行：

1. `pnpm check`：Biome 2 格式、lint 和 import 检查；
2. `pnpm typecheck`：TypeScript 6 strict；
3. `pnpm test`：Vitest 单元/集成测试与 fast-check 性质测试；
4. `pnpm build`：Vite 前端与 Fastify 服务端生产构建；
5. `pnpm test:e2e`：涉及用户主流程、网络边界或数据清除时运行 Playwright。

依赖升级 PR 额外检查：Context7/官方迁移说明、lockfile 差异、构建体积和关键性能指标。普通 CI 不访问真实 chalaoshi。

## 7. 已知未覆盖风险

| 风险 | 现状 | 备注 |
|---|---|---|
| 提示注入防护较弱 | 仅系统提示词声明,无隔离层(D24,已接受) | 架构兜底:摘要任务无状态通道;评测集含注入样本 |
| chalaoshi 合规风险 | Demo 直爬，长期可能建立镜像/向量索引，未取得站方授权(D03/D32,已接受) | 控制措施见 03 §3.4;扩大分发前复审 |
| chalaoshi 域名被墙 | `.de` 需科学上网;备用域名不稳定 | 缓存优先 + 降级常态化;无法保证所有用户可用 |
| zdbk 字段语义未全核验 | `rs`/`yxrs` 等复合字段待选课期样例；学分字段缺失时按 D38 留在待选池 | 核验前相关展示保守处理 |
| 网站无 zdbk 自动接入 | Demo 仅支持手动输入/导入或 fixture | 长期通道见 06 · O-Q11;不得以代登录绕过 |
| 镜像/向量层未完全定型 | 更新频率、存储、embedding 模型和删除 SLA 待决；数据库起步方案已选 pgvector | 不影响 Demo 直爬;见 06 · O-Q9、D33 |
| 精确风险模型未校准 | 志愿与基本顺位已按 D30 闭环，但待选人数为三档聚合值;风险估计仍显示"暂不可评估" | 校准前不得发布数值概率或档位 |
| 性能/兼容/无障碍目标未定 | 浏览器矩阵、性能与 a11y 标准待产品负责人(06 · O-Q6) | 定标后补充测试项 |
| LLM 质量随用户端点浮动 | 能力检测只做门槛,不保证质量上限 | 评测集基于代表性端点,不能覆盖全部 |
