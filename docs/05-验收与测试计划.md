## 3. 求解器正确性测试

### 3.1 性质测试(property tests)

对随机生成的候选/约束输入,任何输出方案必须满足:

1. **无时间冲突**(与固定项、方案内部)——AC-6.3;
2. **池内性**:所有推荐教学班 ∈ 待选池——AC-4.2;
3. **锁定保持**:已选固定、已填志愿锁定、手动锁定不变——AC-6.1/6.2/7.1;
4. **硬约束满足**;无解时给出冲突来源且不放松规则——AC-5.2;
5. **最小扰动**:重新优化的变更集不含锁定项,且不存在"明显更少变更"的等效解(以启发式上界断言);
6. **原子性**:取消/失败路径不留下半成品状态——AC-6.4。

### 3.2 虚拟模型 benchmark(离线仿真)

构建模拟课程/容量/待选人数的虚拟数据集,离线检验:冲突检测、覆盖求解、最小扰动的正确性与性能上限(候选规模压力测试)。

> ⚠️ 边界:在 [02-规则档](02-浙大选课规则核对档.md) 闭环前,benchmark **不得模拟录取过程**(顺位/随机/优先级)——那部分参数就是未核实规则本身。录取仿真属于规则闭环后的风险算法校准工作(06 · O-Q3)。

## 4. LLM 固定评测集

- **偏好结构化**:固定语料("想轻松点""周三留空"等)→ 期望规则栏结构;评 Schema 合规率与语义正确率(人工评分)。
- **评论摘要**:固定评论集 → 必须同时含 pros 与 cons(AC-3.4);低样本标记正确(AC-3.3);含提示注入样本(评论中夹带指令)→ 摘要输出不得包含指令执行迹象,状态零变更(D24)。
- **方案软排序**:固定合法方案集 → `chosenPlanId` 必须在集合内;Schema 失败率统计(D25);排序与偏好一致性抽查。
- **能力检测样题**(D10):作为向导检测的固定题库,含结构化输出与超时用例。

## 5. 端到端与回归

### 5.1 E2E(mock zdbk + fixtures)

- 主流程:首次同意(AC-11.1/11.2)→ 新建 session(AC-1.1)→ 搜课(AC-2.1/2.2)→ 评价卡(AC-3.x)→ 入池(AC-4.1)→ 规则(AC-5.x)→ 生成(AC-6.x)→ 调整/回滚(AC-7.x)→ 定稿/检查(AC-8.1)。
- 降级矩阵:未登录 / 非选课期 / chalaoshi 不可达 / 无 key / 能力检测不通过 / Schema 失败 / 断网(AC-12.1/12.2;AC-10.1)。
- 对话确认制:未确认不改状态(AC-9.1)。
- key 生命周期:配置/掩码/替换/删除(AC-10.2/10.3)。
- **写接口零调用**:E2E 全程网络记录中,zdbk 请求仅命中 [03](03-数据接口契约.md) 只读白名单(AC-8.2)。

### 5.2 zdbk 兼容性回归(D09)

content script 注入后:zdbk 原页面搜索、翻页、选课表单等功能不受影响;样式无泄漏(命名空间检查);zdbk 页面改版时此回归为首要预警。

## 7. 已知未覆盖风险

| 风险 | 现状 | 备注 |
|---|---|---|
| 提示注入防护较弱 | 仅系统提示词声明,无隔离层(D24,已接受) | 架构兜底:摘要任务无状态通道;评测集含注入样本 |
| chalaoshi 合规风险 | 未取得站方授权(D03,已接受) | 控制措施见 03 §3.3 |
| chalaoshi 域名被墙 | `.de` 需科学上网;备用域名不稳定 | 缓存优先 + 降级常态化;无法保证所有用户可用 |
| zdbk 字段语义未全核验 | `rs`/`yxrs`/学分复合字段待选课期样例 | 核验前相关展示保守处理 |
| 录取规则未闭环 | 风险估计"暂不可评估";录取仿真未做 | G0 阻塞发布 |
| 性能/兼容/无障碍目标未定 | 浏览器矩阵、性能与 a11y 标准待产品负责人(06 · O-Q6) | 定标后补充测试项 |
| LLM 质量随用户端点浮动 | 能力检测只做门槛,不保证质量上限 | 评测集基于代表性端点,不能覆盖全部 |
