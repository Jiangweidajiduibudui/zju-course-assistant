## 1. 正式技术方案

项目采用 **TypeScript 单仓库模块化单体**：Vite 构建 React 前端，Fastify 提供同源 REST API 并在生产环境托管静态产物；领域模型和求解器保持框架无关。项目只使用一个 Node.js 运行时、一个 PostgreSQL 数据库和一个 Docker 部署单元。

```text
浏览器：React + TanStack Query + Dexie
                    │ HTTPS / JSON
                    ▼
服务端：Fastify + Zod Schema + 纯 TypeScript 求解器
          │                 │
          ▼                 ▼
 PostgreSQL 公共缓存   chalaoshi 公开资源
```

第一阶段不采用 Next.js、微服务、Redis、独立向量数据库、Python 求解服务或浏览器自动化抓取。

## 3. 仓库结构与依赖方向

```text
src/
├── client/                 # React 页面、features、组件、Dexie、API hooks
├── server/                 # Fastify routes、抓取、repository、服务
├── domain/                 # session、规则、志愿、纯 TS 求解器
└── shared/                 # Zod Schema、DTO、错误码
tests/                      # Vitest / fast-check
e2e/                        # Playwright
sql/migrations/             # 有序、可审查的 PostgreSQL migration
```

依赖方向：`client → shared/domain`，`server → shared/domain`；`domain` 禁止依赖 React、Fastify、Dexie、PostgreSQL 或抓取器；`client` 禁止导入 `server`。

## 4. API 易错点基线

以下规则优先于 AI 训练语料中的旧示例。

### 4.1 Node.js / TypeScript / Vite

- `package.json` 固定 `"type": "module"`；禁止新增 CommonJS `require/module.exports`。
- 客户端 `tsconfig` 使用 `moduleResolution: "bundler"`、`module: "ESNext"`、`noEmit: true`；服务端使用 `module/moduleResolution: "NodeNext"`。
- TypeScript 6 禁止 `moduleResolution: "node" | "node10" | "classic"`、`target: "ES5"` 和 `baseUrl`；路径别名通过 `paths` 与 Vite alias 同步配置。
- Vite 配置使用 `defineConfig`、`@vitejs/plugin-react` 和 `@tailwindcss/vite`；开发期 `/api` 代理只在 `vite dev` 生效，生产由 Fastify 同源提供 API。
- 不使用 Vite experimental API；不依赖 Vite 私有模块路径。

### 4.2 React

- 从 `react-dom/client` 导入 `createRoot`；禁止 `ReactDOM.render`。
- 第一阶段只使用稳定客户端组件和常规 hooks；禁止 React Server Components、Server Actions、canary/experimental API、`experimental_*` 导入。
- TanStack Query 只管理服务端状态；Dexie 是用户持久状态事实源；React state 只保存短生命周期 UI 状态。

### 4.3 Tailwind / shadcn

- Tailwind v4 使用 `@tailwindcss/vite` 和 CSS `@import "tailwindcss"`。
- 禁止 v3 的 `@tailwind base/components/utilities`、`content` 扫描配置和旧 PostCSS 插件写法。
- shadcn 使用 `rsc: false`、Vite 模板和 Tailwind v4；CLI 生成后组件源码归仓库维护。
- `shadcn` 命令执行后必须把新增直接依赖改回精确版本，并重新提交 lockfile。

### 4.4 Fastify / Zod

- Fastify v5 插件只选 Promise 或 callback 一种风格，禁止在 `async` 插件中再接收/调用 `done`。
- `params`、`querystring`、`body`、`response` 必须提供完整对象 Schema；禁止 Fastify v4 的 JSON Schema shorthand。
- 契约以 Zod 4 为单一来源，类型用 `z.infer`，需要 Fastify/LLM JSON Schema 时使用 `z.toJSONSchema()`。
- Zod 使用 `import * as z from "zod"`；禁止引入 Zod 3 专属兼容写法或第三方 JSON Schema 转换器。

### 4.5 TanStack Query / Dexie

- TanStack Query v5 统一使用对象参数：`useQuery({ queryKey, queryFn })`；禁止 v4 及更早的 positional overload。
- 使用 `gcTime`，禁止旧名 `cacheTime`；`queryFn` 必须把收到的 `signal` 传给 `fetch`。
- Dexie 每次结构变化递增数据库版本并保留历史 upgrade 链；禁止直接覆盖已发布的旧 Schema。
- `useLiveQuery` 初始返回 `undefined`，UI 必须显式处理加载态。

### 4.6 PostgreSQL / pg / 缓存

- 普通单条查询优先 `pool.query`；事务必须从 `pool.connect()` 取得同一 client，并在 `finally` 中 `release()`。
- 所有用户或上游输入必须使用 `$1` 等参数化查询；禁止字符串拼接 SQL 值。
- L1 使用 `LRUCache` 命名导入；必须设置 `max` 与 `ttl`。允许过期降级时显式使用 `allowStaleOnFetchRejection`。
- L2 PostgreSQL 至少为 `source_key` 主键和 `expires_at` 建索引；跨实例刷新用 lease/advisory lock，不增加 Redis。
- Cheerio 使用 `import * as cheerio` 并先 `load/loadBuffer`；禁止已移除的静态 `cheerio(...)` API。

### 4.7 测试与质量工具

- Vitest 4 多配置使用 `test.projects`，禁止已弃用的 `workspace`。
- fast-check 异步性质测试必须 `await fc.assert(fc.asyncProperty(...))`；失败日志保留 seed/path 以便复现。
- Playwright 依赖 locator、auto-wait 和 web-first assertions；禁止固定 `sleep` 充当同步。
- Biome 2 使用 `biome ci`；规则预设使用 `preset`，禁止已弃用的 `recommended` 字段。

## 5. Context7 强制工作流

任何新增或修改第三方库相关代码前必须：

1. 从本文件读取固定版本和 Context7 library ID；
2. 用 Context7 查询对应主版本的官方用法和迁移差异；
3. 若 Context7 没有精确 patch，必须确认结果属于同一主版本，并用官方 release notes/npm stable tag 补充核验；
4. 禁止使用 Context7 返回的 experimental/canary 示例；
5. 完成后运行 `pnpm check`、`pnpm typecheck`、`pnpm test`、`pnpm build`；涉及用户流程再运行 E2E。

Context7 是文档检索辅助，不替代 lockfile、类型检查、测试和官方迁移说明。查询不得包含密钥、Cookie、用户数据或项目私密内容。

## 6. 升级策略

- `pnpm-lock.yaml`、Node 版本文件和 Docker image digest 必须提交 Git。
- 不自动跟随 `latest`；稳定版本更新只能通过独立依赖升级 PR。
- 一个 PR 最多升级一个框架族（例如 Vite + plugin-react，或 Tailwind + Vite plugin）。
- 升级前查询 Context7 与官方迁移文档；升级后运行完整 CI、比较构建体积和关键性能指标。
- 业务功能 PR 禁止顺手升级依赖。
- 新增依赖前先证明现有 Web/Node 标准 API 或现有库无法合理完成；不得仅因 AI 建议而引入。

## 7. 性能边界

- 前端：路由/重模块按需加载；长课程列表达到性能阈值后启用虚拟滚动；避免全局大 Context。
- API：L1 `lru-cache` + L2 PostgreSQL；相同抓取 key 使用 single-flight；上游请求必须限频、超时和限响应体。
- 求解器：先 benchmark；典型输入若持续超过 200ms，再迁入 Node worker thread，不提前增加 worker pool 依赖。
- 部署：静态资源使用内容哈希、Brotli/CDN；Fastify 只处理 API 和回源静态托管。
- 不以 Hello World 框架吞吐量作为更换语言或运行时的依据；主要优化目标是请求次数、缓存命中、组合求解和浏览器渲染。
