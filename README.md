# AI 选课助手 · 文档中心

- **文档体系版本**:v2.4
- **项目**:面向浙江大学第一、二轮概率选课的 Web 网站——结合课程/志愿信息、chalaoshi 教师评价数据与用户偏好，用确定性求解器生成可解释的推荐方案;全程 advise-only，网站永不写入选课系统。Demo 后端按需直接抓取 chalaoshi，长期考虑镜像数据层与评论向量索引(D31、D32)。
- **项目状态标签**:开发实施计划已定稿;代码框架与 Task 0 契约/fixture 草案已建立(见 [PROJECT.md](PROJECT.md) §9 脚手架现状),进入模块拆票与并行开发阶段(D33–D42)。

## 3. 阅读顺序建议

- **新加入的开发者**:README → [PROJECT.md](PROJECT.md) → 01(§1–§5)→ 04 → 07 → 08 → 03 → 05。
- **AI 协作会话**:第一份输入固定为 [PROJECT.md](PROJECT.md)(硬规则 + 模块卡片 + 成功判据),再按其阅读协议展开。
- **实现某个功能前**:01 找到对应 FR 与验收标准 → 04 看技术方案 → 07 核对版本/API → 08 查模块边界和任务顺序 → 03 查接口 → 05 查测试映射。
- **想改设计**:先查 06 是否已有决策;推翻已确认决策必须经产品负责人,并在 06 追加新决策条目。
- **涉及志愿/顺位/录取风险**:先读 02——志愿与基本顺位规则已按 D30 闭环，精确风险数值仍冻结。

## 4. 当前关键约束

1. **advise-only 铁律**:网站禁止调用选课、退选、调序等 zdbk 业务写接口(D31)。
2. **规则边界**:第一版只处理第一/二轮概率筛选;志愿模型与基本顺位已确认，但精确概率、风险档位和校准仍冻结，展示一律“暂不可评估”(D01、D30)。
3. **推荐核心 = 集中 `selection-model` + 两阶段 LLM 排序**；硬约束与终校验由确定性模型负责，LLM 只处理组内软排序、Top10 方案比较和解释(D05、D36、D39)。
4. **发布门槛**:规则未核实期间不发布(D06);发布需满足 8 项门槛(D15,测试映射见 [05](docs/05-验收与测试计划.md));开发实施按 [08](docs/08-开发实施计划.md) 的 Task 0–Task 7 门禁推进。
5. **数据与合规**:Demo 后端按需抓取 chalaoshi 并限频/缓存;长期镜像与向量索引须保留来源、更新时间和删除/停止收录机制(D03、D32)。
