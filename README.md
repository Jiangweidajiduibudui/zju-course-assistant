# AI 选课助手 · 文档中心

- **文档体系版本**:v2.2
- **项目**:面向浙江大学第一、二轮概率选课的 Web 网站——结合课程/志愿信息、chalaoshi 教师评价数据与用户偏好，用确定性求解器生成可解释的推荐方案;全程 advise-only，网站永不写入选课系统。Demo 后端按需直接抓取 chalaoshi，长期考虑镜像数据层与评论向量索引(D31、D32)。
- **项目状态标签**:待产品负责人确认(候选:"方案修订中 / 设计定稿但阻塞 / 技术预研阶段",见 [06-决策记录与版本变更.md](docs/06-决策记录与版本变更.md) 开放问题 O-Q1)。

## 3. 阅读顺序建议

- **新加入的开发者**:README → 01(§1–§5)→ 04 → 03 → 05。
- **实现某个功能前**:01 找到对应 FR 与验收标准 → 04 看技术方案 → 03 查接口 → 05 查测试映射。
- **想改设计**:先查 06 是否已有决策;推翻已确认决策必须经产品负责人,并在 06 追加新决策条目。
- **涉及志愿/顺位/录取风险**:先读 02——志愿与基本顺位规则已按 D30 闭环，精确风险数值仍冻结。

## 4. 当前关键约束

1. **advise-only 铁律**:网站禁止调用选课、退选、调序等 zdbk 业务写接口(D31)。
2. **规则边界**:第一版只处理第一/二轮概率筛选;志愿模型与基本顺位已确认，但精确概率、风险档位和校准仍冻结，展示一律“暂不可评估”(D01、D30)。
3. **推荐核心 = 确定性求解器**,LLM 只做模糊语义任务(D05)。
4. **发布门槛**:规则未核实期间不发布(D06);发布需满足 8 项门槛(D15,测试映射见 [05](docs/05-验收与测试计划.md))。
5. **数据与合规**:Demo 后端按需抓取 chalaoshi 并限频/缓存;长期镜像与向量索引须保留来源、更新时间和删除/停止收录机制(D03、D32)。
